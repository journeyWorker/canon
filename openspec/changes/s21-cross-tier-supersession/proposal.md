## Why

The najun-art usability-review synthesis (`target/usage-review/najun-art-dummy/SYNTHESIS.md` B5) and the Plan-agent architecture review (`reviews/plan-agent.json`, `architecture_soundness_score: 8`, this weakness named as its `highest_leverage_improvement` and its single condition for `orchestration_soundness.answer: conditional-yes`) found that **"append-only + deterministic fold" is a GitTier property, not a cross-tier invariant** — even though every load-bearing consumer (canon-gate's byte-identity tests, S3's cursor idempotence, the flywheel folds) presupposes it holds everywhere.

**The correctness gap, grounded in the shipped code:**

- `GitTier` and `R2Tier` are append-only BY CONSTRUCTION: both resolve a record's storage key from `partition::hive_object_key` — `kind={kind}/[area={area}/]{natural_key}__{digest12}.{ext}` (`crates/canon-store/src/partition.rs:165-184`) — so two records with the same natural key but different content resolve to DIFFERENT paths/object keys; a byte-identical resubmission resolves to the SAME key and is a documented no-op. Neither tier can silently overwrite a prior version.
- `PgTier` breaks this: its `records` table is keyed `PRIMARY KEY (kind, id)` (no digest in the key) and every write runs an unconditional `INSERT … ON CONFLICT (kind, id) DO UPDATE SET at = EXCLUDED.at, digest = EXCLUDED.digest, body = EXCLUDED.body` (`crates/canon-store/src/pg_tier.rs:56-79`). This is last-WRITE-wins, not last-`at`-wins: a delayed or reordered write carrying an OLDER `at` unconditionally overwrites a row holding a NEWER `at`, and the overwritten version is gone — no history survives to recover it. `PgTier::read` therefore returns one already-"folded" row per key, folded by write ARRIVAL order, never by `at`.
- Meanwhile `GitTier`/`R2Tier` return every version un-folded, and callers apply `canon-store::fold::fold_latest_by_key` (design D11, `crates/canon-store/src/fold.rs:22-36`) themselves. That fold's own tie-break — "a later-ITERATED item with an equal `at` replaces the earlier one" — is a function of `GitTier`'s scan order, which is unsorted `walkdir::WalkDir` iteration (`crates/canon-store/src/git_tier.rs:140`), itself a function of the host filesystem's directory-entry order — NOT machine-independent.
- s17's own accepted risk (R3, `s17-plan-import/design.md:254-260`) makes same-`at` ties a live scenario: a fresh clone re-stamps file mtimes, and `Task.at`/`Change.at` derive from mtime — so two machines importing the identical plan can legitimately produce records with an IDENTICAL `at`. Combined with the two bugs above, the SAME ledger content can fold to a different "current" answer depending on which machine ran the fold and in what filesystem order — precisely the scale-correctness risk this change closes before routing more load-bearing kinds (`task`, `handoff`, `session`, `run`, `event` — `canon.yaml:38-43`) through PgTier.

**A second, narrower correctness gap in the same trust boundary:** `StalenessCheck`/`ReleaseTrustCheck` each independently call `Utc::now()` inside their own `GateCheck::run` (`crates/canon-gate/src/staleness.rs:186`, `crates/canon-gate/src/trust.rs:199`) and feed it into CEL `age_days` policy predicates (`crates/canon-gate/src/policy.rs:143-157`, whose own `PolicyField::resolve` doc already promises "never read from the wall clock internally" — a promise these two call sites break). `canon-gate`'s acceptance suite pins gate verdicts byte-identical with/without a plan import or a plugin sync (`canon-cli/tests/plugin_sync.rs:174-244`, `plans_ingest.rs:400-425`) — real, valuable tests — but NONE of them exercises a `policy.yaml` with a time-bearing CEL predicate. The one nondeterministic input to the gate authority is exactly the one path its own determinism discipline never checks.

## What Changes

- **PgTier becomes genuinely append-only.** Its `records` table (mutate-in-place, keyed `(kind, id)`) is replaced by an INSERT-only `records_history` table keyed `(kind, id, digest)` — a same-digest resubmission at the same key is a no-op (mirrors `WriteReceipt.deduped`'s existing PgTier semantics, now backed by a table that never loses a version instead of one that overwrites it). `PgTier::read` stops pre-folding: it returns every historical row for a kind (optionally `since`-filtered), the SAME "raw, unfolded, append-only" contract `GitTier`/`R2Tier` already honor — making `Tier::read`'s contract UNIFORM across all three adapters for the first time.
- **`fold_latest_by_key` gets a total, machine-independent tie-break.** Its signature grows a `digest: impl Fn(&T) -> &str` parameter; the winner per key is the item with the greatest `(at, digest)` pair, compared as data — never "whichever the caller happened to iterate last." Every existing call site (`canon-gate::ledger::latest_verdicts`, `staleness.rs::fold_latest_green_cells`, the divergence fold, the flywheel fold) is updated to pass its record's own digest; behavior is unchanged for every existing corpus that has no same-`at` collision, and now deterministic for the ones that do.
- **`GitTier`'s kind-directory scan sorts.** `scan_kind_where`'s `walkdir::WalkDir` traversal adds `.sort_by_file_name()` — deterministic lexicographic order, independent of host filesystem directory-entry order, for every consumer that reads list/iteration order off a `TierReadResult` (not required for the fold's own correctness after the digest tie-break above, but removes a second, independent source of order-dependence).
- **Every PgTier-routed-kind reader adopts the shared fold.** `task`/`handoff`/`session`/`run`/`event` readers (`canon.yaml:38-43` routing) that today trust "one row per key" from `PgTier::read` are updated to apply `fold_latest_by_key` to the (now multi-version) read result — the same step GitTier/R2Tier consumers already perform.
- **The gate authority's one nondeterministic input is injected, not read from the wall clock internally.** `GateContext` grows a `now: DateTime<Utc>` field; `GateContext::load` takes `now` as an explicit parameter instead of never accepting one. `canon gate check`/`canon gate task` (the CLI dispatch boundary, `canon-cli/src/gate.rs`) call `Utc::now()` exactly ONCE per invocation and pass it through — mirroring the ONE-`Utc::now()`-at-dispatch discipline `canon-cli/src/scaffold.rs` already established for `run_scenario_new`/`run_feature_new`. `StalenessCheck`/`ReleaseTrustCheck` read `ctx.now` instead of calling `Utc::now()` themselves; `GateCheck::run`'s own signature is unchanged (`now` travels on `GateContext`, not as a second trait parameter).
- **A time-bearing-policy determinism test closes the coverage gap.** A new acceptance test builds a `policy.yaml` with a CEL `age_days(...)` predicate (staleness or release-trust), a fixed `now`, and asserts the gate report is byte-identical across two independent `GateContext::load` calls at the SAME injected `now` — and, in a companion test, that varying ONLY the injected `now` (all ledger content held fixed) changes the verdict in the expected direction. This is the first test in the suite to exercise a time-bearing policy at all.

### Added Capabilities

- `cross-tier-supersession`: the store-level contract every `Tier` adapter now honors — append-only history (no adapter ever overwrites or discards a prior version in place) plus ONE shared, total, machine-independent fold (`(at, digest)`) every caller applies to resolve "current state" from a tier's raw read result. Covers `PgTier`'s new history table + unfolded read, `fold_latest_by_key`'s digest tie-break, and `GitTier`'s sorted scan.
- `deterministic-gate-clock`: the gate authority's single injected-clock boundary — `GateContext.now`, threaded from ONE `Utc::now()` call at the CLI dispatch layer, consumed by `StalenessCheck`/`ReleaseTrustCheck` instead of either check reading the wall clock itself — plus the time-bearing-policy determinism test that proves it.

### Explicit non-goals

- **No new `RecordKind`, no change to the closed 12-kind set.** `RecordKind::ALL.len() == 12` and its three assertion sites are untouched; this change is a storage/fold-mechanics fix, never a model change.
- **No change to gate/coverage/promotion AUTHORITY.** `canon-gate` still reads nothing importer- or plugin-specific; `canon gate check` verdicts stay byte-identical with and without a plan import or plugin sync (the existing acceptance tests are re-run, not rewritten, as regression proof). Injecting `now` changes WHERE the gate's one time input comes from, never WHAT the staleness/trust-ladder policy logic evaluates.
- **No change to `canon inventory sync`'s single-producer status for `Scenario`.** Untouched by this change.
- **No PgTier schema-migration tooling.** `PgTier::connect`/`connect_live` already run idempotent `CREATE TABLE IF NOT EXISTS` DDL at connect time with no separate migration mechanism in this codebase; this change follows the same convention (a new `records_history` table name, `CREATE TABLE IF NOT EXISTS`-created) rather than introducing schema versioning. A pre-existing local/dev `records` table with real rows is NOT migrated forward automatically — see design.md Risks.
- **No change to `Tier::age`'s own `Utc::now()` call** (`canon-store/src/pg_tier.rs:219`, `r2_tier.rs`'s equivalent, `canon-cli/src/tier.rs:57`). Aging is an operational data-lifecycle sweep against a wall-clock cutoff, not a gate verdict; it is out of scope for the "gate authority" clock-injection half of this change. (It is, however, already a per-row, per-call `Utc::now()` — not part of the cross-tier fold correctness fix either.)
- **No ban on CEL time predicates.** `age_days(...)` stays a supported, pure-given-`now` CEL capability (`canon-policy`'s `evaluate` already never reads the wall clock internally, per its own doc comment) — the fix is injecting the one remaining wall-clock read into the parameter it was always meant to receive, and adding the test coverage that was missing, not removing the feature.
- **No change to `R2Tier`'s write/read shape.** It already shares `GitTier`'s digest-suffixed Hive object-key scheme (`partition::hive_object_key`) and is already append-only by construction; only its downstream consumers gain the same shared-fold treatment PgTier's do, and only where a consumer currently trusts single-row-per-key semantics (`R2Tier` consumers already fold, matching GitTier's contract today).

## Impact

- **`canon-store`**: `pg_tier.rs` (new `records_history` schema/DDL, INSERT-only write, un-folded multi-row read, `age()` adapted to per-row cutoff semantics against the new table), `fold.rs` (`fold_latest_by_key` gains a `digest` parameter and a total `(at, digest)` comparison), `git_tier.rs` (`scan_kind_where`'s `WalkDir` sorted).
- **`canon-gate`**: `context.rs` (`GateContext.now` field, `GateContext::load` takes `now`), `staleness.rs`/`trust.rs` (`ctx.now` replaces internal `Utc::now()`), every in-crate test call site of `GateContext::load` (`context.rs`, `dispatch.rs`, `selftest.rs`, `staleness.rs`, `trust.rs` — roughly 15 sites) passes an explicit fixed `now`.
- **`canon-cli`**: `gate.rs`'s `run_gate_check`/`run_gate_task` each gain the ONE canonical `Utc::now()` call at the dispatch boundary, threaded into `GateContext::load`.
- **Every reader of a `task`/`handoff`/`session`/`run`/`event` PgTier-routed kind**: adopts `fold_latest_by_key` over the (now multi-version) `PgTier::read` result — the same step `GitTier`/`R2Tier` consumers already perform. Exact call-site inventory is design.md's task-list scope; no reader is exempted.
- **No new crate.** `canon-model`/`canon-ingest`/`canon-vocab`/`canon-plugin`/`canon-learn` are unchanged.
