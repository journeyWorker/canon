# Design — s21 cross-tier supersession + deterministic gate clock

## Current state (accurate baseline, verified)

- **`Tier` is one trait, three adapters, with a SILENTLY inconsistent `read` contract.** `crates/canon-store/src/tier.rs`'s own module doc calls this "one storage trait every adapter conforms to … never a tier-specific method reached from outside `canon-store`." In practice `GitTier::read`/`R2Tier::read` return EVERY historical version matching a kind (`git_tier.rs::scan_kind_where` pushes one entry per file under `kind={kind}/`, no dedup by natural key at all); `PgTier::read` returns AT MOST ONE row per `(kind, id)` (`pg_tier.rs:81-87`'s `SELECT … WHERE kind = $1 [AND at >= $2] ORDER BY at`, backed by a table with `PRIMARY KEY (kind, id)`). The trait signature is identical; the semantics are not.
- **GitTier/R2Tier are append-only by construction, not by convention.** Both derive their storage key from `partition::hive_object_key` — `kind={kind}/[area={area}/]{natural_key}__{digest12}.{ext}` (`partition.rs:165-184`) — so two versions of the same natural key with different content necessarily land at different paths/keys; nothing can overwrite a prior version in place. `GitTier::write` treats a same-path (hence same-digest) resubmission as `StoreError::DuplicatePath`, which callers (`persist_idempotent`, s17 design D7) already treat as an idempotent no-op.
- **PgTier is the sole mutate-in-place tier**, by design, for a real reason: its own module doc explains pg holds "in-flight task/handoff state that legitimately mutates in place (a `Handoff`'s `state` column advancing through its closed state machine)." That reason justifies "the CURRENT value of this key can legitimately change"; it does not justify "an arrival with an OLDER `at` can silently clobber a row holding a NEWER one, with the newer version unrecoverable" — which is what the current unconditional `ON CONFLICT (kind, id) DO UPDATE` does (`pg_tier.rs:73-79`).
- **`fold_latest_by_key` (design D11) is the ONE hoisted fold every multi-version reader already shares** — `canon-gate::ledger::latest_verdicts`, `staleness.rs::fold_latest_green_cells`, the divergence fold, the flywheel fold (`fold.rs`'s own module doc: "s15 needs this exact reduction in ≥4 places"). Its current tie-break is documented, explicitly, as iteration-order-dependent: "a tie … is broken by ITERATION order — a later-iterated item … replaces the earlier one" (`fold.rs:14-21`). For `GitTier`-backed callers, "iteration order" traces back to `walkdir::WalkDir::new(&kind_dir).into_iter()` (`git_tier.rs:140`), which is filesystem-directory-entry order — unspecified by POSIX, and empirically NOT byte-stable across filesystems/OSes.
- **s17 already accepted the precondition that makes same-`at` ties real, not theoretical.** `s17-plan-import/design.md` R3: "A fresh clone re-stamps mtimes … writes bodies whose `at` differs" is the ACCEPTED risk for cross-machine DIVERGENCE; the unaccepted, unexamined flip side is that when import timing happens to coincide (batch tooling, CI re-run cadence), two machines CAN produce genuinely identical `at` values for logically-ordered writes, and today's tie-break has no content-derived answer for which one is "current."
- **The gate authority's policy plumbing is ALREADY `now`-parameterized one layer down.** `canon-gate::policy::PolicyField::resolve(&self, record, now: DateTime<Utc>)` and the CEL `evaluate(compiled, record, now, budget)` it calls both take `now` as an explicit argument, with `resolve`'s own doc stating the contract plainly: "`canon_policy::evaluate`'s own contract: never read from the wall clock internally." The bug is one layer UP: `StalenessCheck::run`/`ReleaseTrustCheck::run` (`staleness.rs:184-186`, `trust.rs:196-199`) each independently mint `let now = Utc::now();` and pass THAT into `resolve` — the promised parameter exists, but its value still comes from an internal wall-clock read that changes between two calls a millisecond apart.
- **The CLI already has the "one `Utc::now()` at the dispatch boundary" idiom this fix reuses.** `canon-cli/src/scaffold.rs`'s module doc: "`run_scenario_new`/`run_feature_new` take `at: DateTime<Utc>` as an EXPLICIT parameter — `main.rs`'s dispatch match arms are the ONE place `Utc::now()` is ever called for these commands … so a file this module writes in ONE invocation never straddles two different timestamps." `canon-cli/src/gate.rs`'s `run_gate_check`/`run_gate_task` are the exact analogous dispatch boundary for the gate authority and currently do NOT follow this idiom — they call `GateContext::load(ctx, &registry)` with no clock at all, leaving `Utc::now()` to happen twice, independently, inside two different `GateCheck::run` bodies within the SAME gate invocation.
- **The byte-identity acceptance tests this change must not regress.** `canon-cli/tests/plugin_sync.rs:174-244` and `plans_ingest.rs:400-425` `assert_eq!` gate verdicts with/without a plugin sync or plan import — real regression coverage for connector-never-authority, kept green throughout. Neither exercises a `policy.yaml` with any CEL predicate at all, time-bearing or otherwise — confirmed by grep across `canon-gate`'s test fixtures; this is the exact gap Plan-agent named.

## Architecture — the fold boundary moves from "wherever each tier happens to pre-resolve it" to one shared function every caller applies

```
  write path (any tier)                    read path (any tier)
  ┌─ GitTier::write  ─▶ digest-suffixed path, append-only  ─┐
  ├─ PgTier::write   ─▶ records_history INSERT, (kind,id,digest) key, append-only (NEW) ─┼─▶ Tier::read
  └─ R2Tier::write   ─▶ digest-suffixed object key, append-only  ─┘   returns EVERY version
                                                                        matching kind[, since]
                                                                              │
                                                                              ▼
                                                            fold_latest_by_key(items, key, at, digest)  (fold.rs)
                                                            winner per key = max (at, digest) — pure
                                                            function of content, never iteration order
                                                              │
                                                              ▼
                                                  caller's "current state" (coverage, ledger,
                                                  divergence, flywheel, task/handoff/session/
                                                  run/event readers — ALL tiers, ONE fold)

  gate-authority clock (separate concern, same "inject at the boundary" idiom):

  canon-cli/src/gate.rs (run_gate_check / run_gate_task)
    let now = Utc::now();   // the ONE call, per invocation
        │
        ▼
  GateContext::load(ctx, &registry, now)  ──▶  GateContext { .., now }
        │
        ▼
  StalenessCheck::run(ctx) / ReleaseTrustCheck::run(ctx)  reads  ctx.now
  (no internal Utc::now() left in either check)
```

## Decisions

- **D1 — PgTier becomes append-only via a history table + unconditional-INSERT write, not an at-guarded upsert.** Two shapes were weighed (Plan-agent's own `highest_leverage_improvement` names both): (a) keep `records` mutate-in-place but guard the `UPDATE` with `WHERE (EXCLUDED.at, EXCLUDED.digest) > (records.at, records.digest)` so an out-of-order write is REJECTED instead of silently applied, or (b) replace `records` with an INSERT-only `records_history` table so every version is retained and the fold resolves "current" the same way GitTier's does. **(b) is accepted.** (a) fixes ONLY the ordering bug — an old write can no longer clobber a new one — but does not satisfy "every tier preserves append-only history": the row an old write would have overwritten is still gone the moment a genuinely-newer write lands, and there is no way to recover or audit prior `Handoff`/`Task` states, which the `handoff_templates`/`Handoff::transition_to` state-machine design (S1) treats as meaningful history, not disposable intermediate state. (b) additionally UNIFIES `Tier::read`'s contract across all three adapters for the first time — today's silent divergence (module doc above) is itself a defect independent of the ordering bug. Rejected further alternative: a `canon-store`-level generic "soft-delete + insert" abstraction shared across all three tiers — rejected as scope creep; GitTier/R2Tier need no behavior change at all (they are already correct), so a shared abstraction would touch two adapters that have no bug to fix.
- **D2 — `records_history`'s key is `(kind, id, digest)`, not `(kind, id, at)`.** Two records can legitimately share `(kind, id, at)` (s17 R3's same-mtime cross-machine import) but never share `(kind, id, digest)` unless they are byte-identical — so `(kind, id, digest)` is the correct idempotence key (mirrors `GitTier`'s digest-suffixed PATH exactly) and `INSERT … ON CONFLICT (kind, id, digest) DO NOTHING` gives the SAME "same-digest resubmission is a no-op, `WriteReceipt.deduped = true`" contract PgTier already promises, now backed by a table that cannot lose data instead of one that overwrites it.
- **D3 — `fold_latest_by_key` gains a `digest: impl Fn(&T) -> &str` parameter; the tie-break becomes `max_by_key(|item| (at(item), digest(item)))`.** This is a pure, total, machine-independent order: for any two items with equal `at`, the lexicographically-greater digest wins, deterministically, regardless of which one the caller happened to construct or iterate first. Every existing caller already has SOME digest available for its item (either the item carries `partition::content_digest12` over its own raw JSON, or — for PgTier/R2Tier rows — the persisted `digest` column read back verbatim); no caller needs to invent or recompute a hash it does not already have. Rejected alternative: keep `at`-only comparison and instead REJECT same-`at` writes at write time across all tiers — rejected because `at` collisions are a legitimate, accepted cross-machine artifact (s17 R3) the fold must resolve, not an error condition a write-time check can meaningfully refuse (a git-tier write has no visibility into a pg-tier write's `at`, and vice versa, so no single write path can enforce global `at` uniqueness even if it wanted to).
- **D4 — `GitTier::scan_kind_where` sorts its `WalkDir` traversal (`.sort_by_file_name()`).** Belt-and-suspenders, not a correctness requirement once D3 lands (the fold no longer depends on iteration order for CORRECTNESS) — but `TierReadResult.records`' iteration order still leaks into human-facing output (violation listing order, CLI summaries) and other order-sensitive consumers that predate D3; making the SCAN itself deterministic removes a second, independent source of host-filesystem-dependent behavior rather than relying solely on every downstream consumer remembering to fold. Cost: one `.sort_by_file_name()` call; no behavior change for any consumer that already folds.
- **D5 — Every PgTier-routed-kind reader is migrated to call the shared fold; `PgTier::read`'s new "raw history" contract is NOT hidden behind a second, pre-folding read method.** Rejected alternative: give `PgTier` two read methods — `read_history` (new, raw) and `read` (old name, pre-folds internally so existing callers need zero changes). Rejected because it recreates exactly the defect D1 exists to close: a caller reading `task`/`handoff`/`session`/`run`/`event` state through `Tier::read` would AGAIN see a tier-specific "already resolved for me" contract, silently different from what the SAME trait method returns for `git`/`r2`-routed kinds — the "one storage trait, three conforming adapters" doc comment would still be false. The one-time migration cost (D5's task-list scope: every reader of a pg-routed kind gains a `fold_latest_by_key` call) is accepted as the honest price of the trait actually meaning one thing.
- **D6 — `GateContext` carries `now: DateTime<Utc>`; `GateContext::load` takes it as a parameter; `GateCheck::run`'s signature is unchanged.** Two shapes were weighed: thread `now` through `GateCheck::run(&self, ctx, now)` directly, or carry it on `GateContext` (already the "everything a wave-2 check needs" bundle, per its own module doc) and leave `run`'s signature alone. **The `GateContext` field is accepted.** `GateCheck` is a `Send + Sync` trait object collection iterated once per gate run (`dispatch::check_set`); adding a second parameter to its one trait method would touch every implementor (`CoverageCheck`, `LedgerCheck`, `StalenessCheck`, `TrustLadderCheck`, `ReleaseTrustCheck`) for a value exactly TWO of them use, and would diverge from the established "`GateContext` is everything a check needs" contract the trait's own doc states. Carrying it on `GateContext` means the checks that do not need a clock (coverage, ledger, trust-ladder-without-release) are untouched.
- **D7 — Inject the clock; do not ban time predicates.** `age_days(...)` CEL predicates stay supported: `canon-policy::evaluate` is ALREADY pure-given-`now` (module doc; the promise the two checks were breaking is about WHO calls it, not whether the function itself is deterministic). Banning time predicates would delete a working, already-tested-pure capability to work around a bug in exactly two call sites — a bigger behavior change for a smaller fix. Rejected because it throws away real, wanted policy expressiveness (a release policy legitimately wanting "flag evidence older than N days" is a reasonable, already-designed-for CEL predicate) to avoid writing 15 call-site updates.
- **D8 — `Tier::age`'s own `Utc::now()` (`pg_tier.rs:219`, `r2_tier.rs`'s equivalent, `canon-cli/src/tier.rs:57`) is explicitly OUT of scope.** It drives an operational data-lifecycle sweep (move records older than a configured threshold to colder storage) rather than a gate VERDICT; injecting it would widen this change's surface without closing the correctness gap Plan-agent named (gate-authority nondeterminism), and aging is already idempotent under a re-run regardless of exactly which instant `Utc::now()` returns (tier-adapter-trait spec's own "aging is idempotent under a duplicate run" contract, unaffected by clock jitter at aging granularity).

## Risks

- **R1 — PgTier schema-name change is a breaking DDL change for any existing local/dev pg volume.** `PgTier::connect`/`connect_live` run `CREATE TABLE IF NOT EXISTS {schema}.records_history …` — a fresh table name, never touching a pre-existing `{schema}.records` table left behind by a prior canon version. A developer with dev-stage data already written to the old `records` table sees an EMPTY `records_history` on first connect after this change (no automatic backfill). Accepted: this codebase has no schema-migration mechanism today (verified: `connect`'s DDL is `CREATE TABLE IF NOT EXISTS`, idempotent-forward-only, no version table), the only known live-pg surface is `tests/pg_tier_live.rs` behind the `live-pg` feature against the repo's own `docker-compose.yml` stack (disposable, no persisted production data), and the change proposal is explicit (non-goals) that migration tooling is out of scope. Mitigation: this change's task list drops the old `records` table's `CREATE TABLE IF NOT EXISTS`/`CREATE INDEX` DDL entirely rather than leaving it as dead code beside the new table, so there is no ambiguity about which table is authoritative going forward.
- **R2 — `records_history` grows unboundedly for a hot, frequently-updated key (e.g. a `Handoff` cycling through several states in one session) without the existing `age()` sweep changing shape.** Accepted: `age()`'s per-row `at < cutoff` predicate (unchanged by this design beyond operating against the new table) already ages every row independently of whether it was ever the "current" version for its key — a superseded `Handoff` state ages to `r2` on the SAME schedule its replacement will, once IT ages past the cutoff too. This is a storage-cost/query-cost tradeoff already accepted for `GitTier` (which has always retained every version); PgTier now inherits the identical tradeoff instead of the unlimited-history GitTier already lives with, not a NEW risk this change introduces.
- **R3 — Migrating every PgTier-routed-kind reader to call the shared fold is real work with real blast radius (D5).** `task`/`handoff`/`session`/`run`/`event` readers across `canon-cli`/S9 marts/`canon query` that currently assume "one row per key" from `PgTier::read` must each be found and updated, or they silently start seeing every historical version as if it were N independent current records (a correctness regression WORSE than the bug being fixed, if missed). Mitigation: task-list scope treats "grep every `PgTier`/`TierQuery`-based reader for a pg-routed kind, confirm it already folds or add the fold" as an explicit, checked task (not "as needed"), and an acceptance test asserts row-COUNT parity — a reader that used to see exactly 1 row per key for an unchanged corpus still sees exactly 1 (folded) row per key after this change, only now correctly ordered rather than accidentally so.
- **R4 — `fold_latest_by_key`'s signature change touches every existing call site in one commit.** A caller that forgets to update its closure list fails to COMPILE (Rust's `impl Fn` parameter is not optional), so this is a fail-loud, not fail-silent, migration risk — the compiler enumerates every site that needs the new `digest` closure, and `cargo build --workspace` either finds all of them or the change does not build. Explicitly not a runtime risk.
- **R5 — a time-bearing policy test that hardcodes a specific `now` could itself become a silent bit-rot risk if `age_days`'s CEL semantics ever change.** Mitigated by anchoring the test's fixed `now` and evidence `at` values as EXPLICIT UTC constants (not `Utc::now() - Duration::days(N)` computed at test-run time — the exact anti-pattern this change fixes in production code) so the test's own expected verdict is reproducible and reviewable, not itself wall-clock-dependent.

## Sequencing

- **P1 — `fold_latest_by_key`'s digest tie-break (D3), standalone.** No caller migration yet beyond passing a digest closure at each existing call site (mechanical, compiler-enforced per R4). Ships alone; `GitTier`/`R2Tier` consumers get the machine-independent tie-break immediately with zero PgTier changes.
- **P2 — `GitTier` sorted scan (D4).** Independent of P1/P3; can land in either order.
- **P3 — PgTier `records_history` (D1/D2), after P1.** The new table + INSERT-only write + un-folded read; depends on P1 so the fold this unlocks already has its total order.
- **P4 — PgTier-routed-kind reader migration (D5), after P3.** The blast-radius item (R3); every reader of `task`/`handoff`/`session`/`run`/`event` state adopts the shared fold. Gated behind P3 because there is nothing to migrate readers ONTO until `records_history` exists.
- **P5 — gate-authority clock injection (D6/D7), independent of P1-P4.** `GateContext.now`, the CLI dispatch-boundary `Utc::now()`, `StalenessCheck`/`ReleaseTrustCheck` reading `ctx.now`, and every in-crate test call site of `GateContext::load` passing an explicit `now`. Can land before, after, or interleaved with P1-P4 — it shares no code with the tier/fold work beyond both living under "gate-authority determinism."
- **P6 — the time-bearing-policy determinism test (after P5).** Requires the injected clock to exist before it can assert on it.
- **P7 — closure.** Re-run (not rewrite) the existing byte-identity acceptance tests (`plugin_sync.rs:174-244`, `plans_ingest.rs:400-425`) as regression proof that P1-P6 changed WHERE inputs come from without changing WHAT the gate decides; `RecordKind::ALL.len() == 12` re-asserted at its three sites; `cargo test --workspace` full-suite green.
