# Design — s24 scope-status surface

## Current state (accurate baseline, verified)

- **`mart_scope_status` (`crates/canon-store/sql/views.sql:271-287`) is real, tested, and unreachable from any blessed CLI surface.** It joins `int_task_scenario_refs` (a `Task`'s declared `[covers: …]` scenario refs, s20) against `mart_trust_matrix` (evidence-presence `covered`/`green`, keyed `task_id`) and `porting.coverage` overlay rows (spec-authorship `covered`, keyed `scenario_id`), producing one row per declared `(task_id, scenario_id)` pair with columns `task_id, scenario_id, task_status, evidence_covered, green, spec_covered`. `crates/canon-store/tests/e2e_scope_status_duckdb.rs` proves the join correct via a real `duckdb` CLI run. Neither `canon report` (5 hardcoded panels, `crates/canon-report/src/render.rs`) nor `canon report --snapshot` (`SNAPSHOT_TABLES`, `crates/canon-report/src/snapshot.rs:24-25`, also hardcoded to 5) expose it.
- **`canon-report`'s five S9-owned panels are a fixed, parallel structure across three files**, all touched by any new panel: `crates/canon-report/src/marts.rs` (one `fetch_*` per panel, a thin `SELECT * FROM <view> ORDER BY <cols>` plus a `&'static [&'static str]` column-order constant — module doc: "No aggregation happens here … design D1"), `crates/canon-report/src/render.rs` (`ReportMarts` struct + `render()`'s sequential `## <Heading>` + prose + `render_table` blocks), `crates/canon-report/src/snapshot.rs` (`SNAPSHOT_TABLES`, a bare name list consumed generically by `snapshot()`'s per-table `COPY` loop — no per-table special-casing exists to update). `crates/canon-report/src/lib.rs::report()` is the one call site that constructs a `ReportMarts` from the five `fetch_*` calls.
- **`canon report` already tolerates a git-only repo for its existing 5 panels — via Rust-side seeding, not a SQL fix.** `crates/canon-report/src/roots.rs::Roots::ensure_seeded` writes one zero-row placeholder parquet file per `read_parquet`-backed source (`r2_root`; `learn_root/strategies`; `learn_root/trajectories`) at the exact directory depth each view's glob expects, called before every `duckdb -init` invocation (`crates/canon-report/src/query.rs`'s module doc). `mart_scope_status` itself never reads `read_parquet` directly (it joins `int_task_scenario_refs`, `mart_trust_matrix`, and a `stg_records`-filtered `porting.coverage` subquery — all ultimately over `stg_records`, which unions `stg_git_records` (`read_text`, tolerant of zero matches) and `stg_r2_records` (`read_parquet`, the one this seeding step protects)). Because `Roots::ensure_seeded` already runs unconditionally before *every* `canon report` query today, the new panel inherits this tolerance automatically — no new seeding logic is needed for `canon report` itself. `crates/canon-report/tests/fresh_repo.rs` already proves the general case (a completely fresh repo, all 5 existing panels render empty, zero crash); this change's task list extends that same test to assert 6 empty panels instead of 5, the acceptance proof that the new panel inherits the same tolerance rather than needing its own fix.
- **The raw `duckdb -init sql/views.sql` fallback (documented in `.claude/skills/tiered-storage/SKILL.md`) has no equivalent seeding step** — it is a bare `duckdb` CLI invocation the operator runs directly, never routed through `canon-report`'s Rust code, so it never benefits from `Roots::ensure_seeded`. This is the mechanism behind the round-2 finding: `canon report` (Rust-mediated) already works on a git-only repo; raw `duckdb -init` (unmediated) does not, for ANY mart including the 5 that already exist today — this is a pre-existing condition of the raw fallback, not something `mart_scope_status` introduces.

## Decisions

- **D1 — The new panel's column set is exactly `mart_scope_status`'s own `SELECT` list, unmodified.** `task_id, scenario_id, task_status, evidence_covered, green, spec_covered` — no Rust-side renaming, filtering, or derived column, mirroring every existing panel's `fetch_*` (design D1: "no second aggregation layer," `canon-report` renders/exports what the DuckDB view already computed).
- **D2 — The new panel is appended LAST (after `## Review burn-down`), and `mart_scope_status` is appended LAST in `SNAPSHOT_TABLES`.** Two placements were weighed: insert it second (directly after `## Trust matrix`, since it is a `Task`-centric refinement of that same panel — the natural reading order for a planner) or append it last. **Append-last is accepted.** Every existing panel's markdown output for a fixture that predates this change must stay byte-for-byte identical (no existing consumer's diff/golden output should shift merely because a panel was inserted ahead of it), and `SNAPSHOT_TABLES`'/`manifest.json`'s existing 5-entry `(table, file)` order is itself asserted by `tests/snapshot.rs::EXPECTED_CONTRACT` — appending avoids reordering anything already covered by that contract test. The cost (a planner reads one panel further down) is minor and consistent with how `mart_scope_status` itself was added last among the marts in `views.sql`.
- **D3 — `ORDER BY task_id, scenario_id`, matching the view's own `ORDER BY r.task_id, r.scenario_id`.** Every existing panel's `fetch_*` re-states the view's own natural order explicitly (design comment on `marts.rs`: "for stable markdown rendering"); this panel follows the identical convention rather than introducing a new one.
- **D4 — Raw `duckdb -init sql/views.sql` git-only tolerance is a named non-goal, not attempted.** Three shapes were weighed: (a) restructure `stg_r2_records`/`stg_strategy_items`/`stg_trajectories` in `views.sql` to `UNION ALL` a static-schema, zero-row literal `SELECT` alongside each `read_parquet` call so an empty glob degrades gracefully even without Rust-side pre-seeding; (b) leave `views.sql` untouched and instead document the existing manual-seed workaround (already used by `crates/canon-store/tests/e2e_scope_status_duckdb.rs`) in the tiered-storage skill docs; (c) do neither, scope it out entirely. **(c) is accepted for this change.** (a) requires hand-maintaining each `read_parquet` source's exact Arrow-equivalent schema a second time, in SQL, independently of `crates/canon-report/src/roots.rs`'s `r2_schema()`/`learn_store_schema()` Rust functions that already encode it — two schemas for the same three sources is a drift risk `views.sql`'s own module doc explicitly does not currently carry, and DuckDB's `UNION ALL` schema-reconciliation behavior across a real `read_parquet` result and a hand-typed literal `SELECT` is not proven safe for every DuckDB version this codebase supports. (b) is a legitimate lower-risk follow-up but touches documentation, not a spec-worthy behavior change, and is not required to satisfy the round-2 finding, which asks for `mart_scope_status` to be reachable from a *blessed* surface — `canon report` — not for the unblessed raw-`duckdb` fallback to match it. Accepting (c) keeps this change scoped to `canon-report`, with zero `views.sql` risk.

## Risks

- **R1 — `SNAPSHOT_TABLES` growing from 5 to 6 entries is a breaking change for any external consumer that hardcodes "5 parquet files" or a fixed `manifest.json` table order.** In-repo, `crates/canon-report/tests/snapshot.rs`'s own `EXPECTED_CONTRACT` and several `assert_eq!(…, 5, …)`/`assert_eq!(…, 6, …)`-shaped assertions hardcode the count — these are updated as part of this change's task list, not left to bit-rot. Outside this repo, `manifest.json` is self-describing (`tables: [{table, file}]`, no fixed-length assumption baked into its own JSON shape) — a consumer that iterates `manifest.tables` rather than assuming a fixed count is unaffected; a consumer that hardcodes the count is, by definition, coupled to an internal implementation detail this codebase makes no external stability promise about (no versioned/public API contract exists for `SNAPSHOT_TABLES`'s cardinality). Accepted, no back-compat shim — matches this codebase's established clean-cutover convention.
- **R2 — `evidence_covered`/`green`/`spec_covered` can each independently be `NULL`** (the view's own `LEFT JOIN`s: a task with no evidence yet, or a scenario with no `porting.coverage` overlay row, or a repo with the `porting` plugin never installed at all). This is NOT a new risk this change introduces — `mart_trust_matrix`'s existing `covered`/`green` columns already flow through the same `render_table`/`cell()` NULL-handling every existing panel relies on (`crates/canon-report/src/render.rs::cell`, already generic over `Row::get`'s `Option`). No new NULL-handling code is needed; the acceptance test (task list) asserts a NULL cell renders the SAME placeholder existing NULL cells (e.g. `mart_trust_matrix.who` for an uncovered task) already render.
- **R3 — A repo with zero `Task.scenario_refs` declarations (the pre-s20 majority case, and every fixture that predates this change) renders an empty `## Scope status` panel.** Accepted, matches every other panel's documented "no rows" behavior — an empty panel is informative (nothing declared to unify yet), not an error state; `tests/fresh_repo.rs`'s extended assertion (5 → 6 `_No rows._` panels) is the direct regression proof.

## Sequencing

- **P1 — `marts.rs`: `SCOPE_STATUS_COLUMNS` + `fetch_scope_status` (D1, D3).** Standalone; no dependency on the render/snapshot layers.
- **P2 — `render.rs`: `ReportMarts.scope_status` field + `## Scope status` render block (D2), after P1.** Needs `fetch_scope_status`'s `MartResult` shape to exist first.
- **P3 — `lib.rs`: wire `fetch_scope_status` into `report()`'s `ReportMarts` construction, after P1/P2.**
- **P4 — `snapshot.rs`: `SNAPSHOT_TABLES` gains `mart_scope_status` (D2), independent of P1-P3** (the snapshot export loop is generic over any view name — no dependency on the markdown-rendering path at all).
- **P5 — Fixture + test updates, after P1-P4.** `fixtures/corpus.rs` extends with a `Task.scenario_refs` + `porting.coverage` row; `tests/marts.rs`, `tests/byte_stability.rs`, `tests/fresh_repo.rs`, `tests/snapshot.rs` all updated/extended.
- **P6 — Closure.** `bunx openspec validate --strict s24-scope-status-surface`; workspace build/test/clippy green; `canon gate check` byte-identity re-confirmed unaffected (no `canon-gate` file touched by this change at all, so this is a pure regression guard, not a code-path this change exercises).
