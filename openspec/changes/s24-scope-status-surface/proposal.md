## Why

The eno-drift round-2 usability re-review (`target/usage-review/eno-drift/SYNTHESIS-ROUND2.md`, gap #3) and all three read-surface personas independently confirmed the same operational gap:

- **Planner** (`reviews/planner.json`, `score_justification`): "my #1 ask — 'is this scope DONE and VERIFIED in one query' — is still not answerable from a CLI a planner would actually run: the Task↔Scenario join is real in the ledger now, but neither blessed read surface (`canon report`'s 5 fixed panels, `canon report --snapshot`'s hardcoded `SNAPSHOT_TABLES`) exposes `mart_scope_status`." The same review names the raw-`duckdb` escape hatch as having "NO operator workaround, not even a documented one: pointing `CANON_R2_ROOT`/`CANON_LEARN_ROOT` at an empty directory … still hard-errors on the zero-match `read_parquet` glob."
- **Developer** (`reviews/developer.json`): "s20's `mart_scope_status` view … is real and correctly-scoped … but reaches a developer through NEITHER blessed surface: `canon report --snapshot` exports exactly 5 marts … scope_status is absent — and the documented raw-duckdb fallback (`duckdb -init sql/views.sql`) hard-errors immediately on eno-drift's git-only [repo]." Severity: "medium — the s20 join is implemented and tested but operationally undeliverable to a git-only daily driver."
- **Plan-agent** (`reviews/plan-agent.json`): "the join now EXISTS (B3 genuinely closed at the model/ledger level) but is not yet an answerable canon query, which is a surface gap, not an architecture regression."

**Grounded in the shipped code:**

- `crates/canon-store/sql/views.sql:271` defines `mart_scope_status` — one row per declared `(task_id, scenario_id)` pair, unifying `task_status` (done, the checkbox) × `evidence_covered`/`green` (verified, evidence-side) × `spec_covered` (scenario-authored, spec-side). Its own comment states the intended posture explicitly: "Read-only reporting ONLY — never a `canon-gate` input; `canon gate check` verdicts are byte-identical before and after this view exists."
- `crates/canon-report/src/snapshot.rs:24-25`'s `SNAPSHOT_TABLES` constant is a hardcoded 5-element list (`mart_trust_matrix`, `mart_session_costs`, `mart_role_memory`, `mart_flywheel_funnel`, `mart_review_burndown`) that does not include `mart_scope_status`; `crates/canon-report/src/render.rs`'s `ReportMarts` struct and `render()` function are equally hardcoded to the same 5 panels. Neither `canon report` nor `canon report --snapshot` can surface the view at all today.
- Raw `duckdb -init crates/canon-store/sql/views.sql` (documented in `.claude/skills/tiered-storage/SKILL.md` and its `.codex`/`canon/skills` mirrors as the manual escape hatch) hard-errors on a git-only repo: `stg_r2_records` (`views.sql:70-77`) is `read_parquet(getenv('CANON_R2_ROOT') || '/kind=*/**/*.parquet')`, and DuckDB's `read_parquet` hard-errors when its glob matches zero files — true even when `CANON_R2_ROOT` points at an existing-but-empty directory, since there is still no file to bind the glob against.
- **`canon report` already solves exactly this problem, but only for itself, in Rust, not in SQL.** `crates/canon-report/src/roots.rs` (`Roots::ensure_seeded`) writes one zero-row placeholder parquet file per `read_parquet`-backed source (r2, learn/strategies, learn/trajectories) at the *exact* glob depth each view expects, before every `duckdb -init` invocation (`crates/canon-report/src/query.rs`'s own module doc: "roots seeded first … so an empty r2/learn source never aborts an unrelated query"). `crates/canon-report/tests/fresh_repo.rs` proves this live: a completely fresh, git-only repo renders all 5 existing panels successfully via `canon report`. This seeding step lives entirely in `canon-report`'s Rust code — a raw `duckdb -init sql/views.sql` invocation never runs it, which is why the documented manual fallback still hard-errors while `canon report` itself does not.

The join layer (s20) is done; the surface layer is not. This change closes the surface gap for the blessed CLI (`canon report`), and explicitly scopes out re-implementing the Rust-side seeding trick inside `views.sql` itself for the raw-`duckdb` escape hatch (see non-goals).

## What Changes

- **`canon-report`'s panel set grows a sixth panel, `## Scope status`, sourced from `mart_scope_status`.** `crates/canon-report/src/marts.rs` gains `SCOPE_STATUS_COLUMNS` (`task_id`, `scenario_id`, `task_status`, `evidence_covered`, `green`, `spec_covered` — the view's own `SELECT` list, unchanged, per design D1's "no second aggregation layer") and `fetch_scope_status`, ordered `ORDER BY task_id, scenario_id` (mirrors the view's own `ORDER BY`). `crates/canon-report/src/render.rs`'s `ReportMarts` struct gains a `scope_status: MartResult` field, appended last (existing 5-panel order and every existing panel's markdown is untouched, byte-for-byte, for any fixture that predates this change); `render()` gains one more `render_table` call under a `## Scope status` heading, following the same "one panel = one heading + one prose line + one table" shape every existing panel uses. `crates/canon-report/src/lib.rs`'s `report()` wires `marts::fetch_scope_status` into the new field.
- **`SNAPSHOT_TABLES` gains `mart_scope_status`, appended last.** `crates/canon-report/src/snapshot.rs:24-25`'s list grows from 5 to 6 entries; `canon report --snapshot <dir>` now writes `mart_scope_status.parquet` and lists it in `manifest.json`'s `tables` array alongside the existing 5, with no change to the export mechanism (`export_view`'s generic `COPY "<view>" TO '<path>' (FORMAT parquet)` already works for any view name).
- **Both new panels render deterministically**, matching every existing panel's discipline: timestamp-free (no `generated_at`/wall-clock content anywhere in the rendered table), rows pre-sorted by the view's own `ORDER BY` (never a `HashMap`/unordered iteration), and an explicit `_No rows._` placeholder when the view returns zero rows (the same `render_table` helper every panel already shares — no new empty-state branch needed).
- **The raw-`duckdb -init sql/views.sql` git-only hard-error is explicitly NOT fixed by this change** (see non-goals) — `canon report` is named the blessed surface for `mart_scope_status` (and every other mart), not the raw-DuckDB fallback.

### Added Capabilities

- `scope-status-report`: `canon report`'s new sixth panel (`## Scope status`, sourced from `mart_scope_status`) plus `canon report --snapshot`'s `SNAPSHOT_TABLES` growing to include it — the "is this scope DONE (task checkbox) and VERIFIED (evidence-covered) and SPEC-COVERED (scenario-authored)" answer, in one query, from the one blessed CLI surface, on a git-only repo, with no configured r2/learn tier required.

### Explicit non-goals

- **No change to `crates/canon-store/sql/views.sql`'s `stg_r2_records`/`stg_strategy_items`/`stg_trajectories` `read_parquet` glob tolerance.** Making a raw `duckdb -init sql/views.sql` invocation itself tolerate a zero-file glob (without `canon-report`'s Rust-side `Roots::ensure_seeded` pre-seeding step running first) would require either (a) re-deriving `Roots::ensure_seeded`'s exact per-source glob-depth/schema knowledge a second time as a static-schema empty-relation `UNION ALL` inside the SQL file itself — duplicating that knowledge in two places (a DuckDB `arrow`-backed Rust schema and a hand-written SQL schema) with a real drift risk if either changes independently — or (b) changing DuckDB's own `read_parquet` zero-match behavior, which is not this codebase's to change. `canon report` is the blessed surface for `mart_scope_status` (and every other mart); this change closes the gap there. The raw-`duckdb -init` path (documented in `.claude/skills/tiered-storage/SKILL.md` and its mirrors) remains an unblessed, power-user escape hatch that already has a known, established workaround for anyone who chooses to use it directly — manually seed one throwaway file per empty `read_parquet` root at the correct glob depth, exactly the pattern `crates/canon-store/tests/e2e_scope_status_duckdb.rs::seed_empty_r2_root`/`seed_empty_learn_root` already use for test fixtures — but formalizing or automating that workaround for interactive/manual use is out of scope for this change.
- **No new `RecordKind`, no change to the closed 12-kind set.** `RecordKind::ALL.len() == 12` and its three assertion sites are untouched; this change is a reporting-surface addition, never a model change.
- **No change to `mart_scope_status`'s own SQL definition, join semantics, or the s20 `task-scenario-join` capability it belongs to.** The view (`views.sql:271-287`) is consumed exactly as `task-scenario-join` already shipped it — no new column, no new join, no re-derivation.
- **No change to gate/coverage/promotion authority.** `canon-gate` reads nothing importer- or plugin-specific and nothing from `canon-report`; `canon gate check` verdicts are unaffected — the scope-status panel is read-only reporting, consumed by nobody but a human or dashboard reading `canon report`'s output, exactly the posture the view's own SQL comment already declares ("never a `canon-gate` input").
- **No change to `canon inventory sync`'s single-producer status for `Scenario`.** Untouched by this change.
- **No CLI flag changes.** `canon report`/`canon report --check`/`canon report --snapshot` keep their existing argument shapes (`crates/canon-cli/src/report.rs`); the new panel/export entry appears automatically, with no new opt-in/opt-out flag — matching every existing panel's non-optional inclusion.

- **`canon-report` tests**: `tests/byte_stability.rs` (unaffected assertions still hold; report still starts `# canon report\n`, still timestamp-free), `tests/fresh_repo.rs` (the "`_No rows._` count" assertion moves from 5 to 6, still zero-crash on a completely fresh repo — proving the new panel inherits the SAME `Roots::ensure_seeded` git-only tolerance every existing panel already has, at no extra cost), `tests/marts.rs` (new `scope_status_matches_the_fixture_corpus_exactly` test), `tests/snapshot.rs` (`EXPECTED_CONTRACT`/table-count assertions move from 5 to 6), `fixtures/corpus.rs` (extends the shared fixture with a `Task.scenario_refs` declaration + `porting.coverage` overlay row so the new panel has non-empty, known-value rows to assert against).
