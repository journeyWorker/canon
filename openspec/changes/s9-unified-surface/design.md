## Context

Design doc §5 S9 (`docs/superpowers/specs/2026-07-10-canon-design.md:318-329`):
`canon report` (markdown, generated never hand-edited, D16) + dashboard: DuckDB-Wasm
over a parquet snapshot with a freshness banner — mirrors `canon-report`'s numbers,
never re-derives (spec-status-dashboard pattern). Panels: change/task trust matrix
(covered × green × who), session costs (by role/repo/session), role memory
(strategies, hit rate, effect), flywheel health (verdicts→distilled→retrieved→applied
funnel), review-feedback burn-down. Also wires the donor monorepo's orphaned `GET /api/openspec/
changes` rollup as one data source. Acceptance: dashboard renders all panels from a
fixture snapshot; `canon report --check` is byte-stable on unchanged input.

Two donor lineages this design lifts from (§3):

- **The donor parity harness** (D16 "Derived-only status: no hand-authored aggregates,
  drift-checked reports" — `spec/` holds inputs only; the matrix/verdicts/rollups are
  generated by its `report` command; committing the report means CI regenerates
  and fails on diff; the header embeds input digests). Its `cmd_report` is the
  literal `--check` shape: read the existing file, compare byte-for-byte against
  freshly generated content, "no drift" on match (exit 0), "DRIFT" with the
  stale-file path on mismatch (exit 1), "MISSING" when the report was never generated.
- **The donor consumer repo's `spec-status-dashboard` change**: D2 (precomputed
  Parquet snapshot loaded into an in-memory DuckDB, because a browser sandbox cannot
  read arbitrary repo files, and Parquet — not the native `.duckdb` binary — is the
  version-stable cross-engine interchange between the CLI writer and any Wasm
  reader); D7 (the app queries the snapshot only, never re-derives a mart, so a
  mart-logic change is a single-place edit + re-export and the app is unaffected);
  a `manifest.json` (`generated_at`, `source_git_sha`, `source_duckdb_sha256`,
  `tables: [{table, file}]`) is the **declared** table→file map the loader iterates,
  because a Wasm app cannot enumerate an asset directory at runtime; the freshness
  banner surfaces that provenance so "matches the report" is scoped to the recorded
  snapshot input, not the live checkout.

the donor monorepo's `GET /api/openspec/changes` already implements the
cross-worktree rollup: it requires an `X-User-Id` header (401 without it),
delegates to the donor CLI's `walkWorktrees()` (60s TTL cache keyed by cwd), and
returns `{ changes: RolledChange[] }` where `RolledChange` is `{ worktree, branch,
slug, route, created, proposalTitle }`. Confirmed orphaned: only its own route test
and a Bruno request collection reference it; no dashboard frontend code calls it.

Prerequisites from earlier waves (S9 is W3, after W0's S0→S1→S2 and W1/W2): S1's
`canon-model` supplies the versioned record types + join spine (`Change`, `Task`,
`Session`, `Review`, `EvidenceRecord`, …) this report aggregates; S2's `canon-store`
already ships the `stg_/int_/mart_` DuckDB-view layer over parquet + git files
(specdb.sql pattern) that S9's new dashboard-specific marts sit on top of; S3/S4 feed
session and review/handoff data into those marts; S5 computes the trust ladder;
S6/S7/S8 produce the strategy/reward/retrieval events the role-memory and flywheel
panels read.

## Goals / Non-Goals

**Goals:**

- `canon report`'s markdown output is the single authoritative status surface;
  nothing hand-typed (a prose progress file, a manually updated table) may claim
  authority over it (D16).
- `canon report --check` is byte-stable: identical input (same corpus/policy/
  ledger-head digest) produces byte-identical output on every run, so CI can gate on
  drift exactly like `parity.py report --check`.
- The dashboard's numbers can **never** contradict `canon report`'s, because it has
  no mart-derivation code of its own — only thin SELECT/filter over the snapshot
  `canon report --snapshot` produced.
- All five panels render end-to-end from one fixture snapshot (no live tier access
  required to develop or test the dashboard).
- The dashboard runs with **zero network** after the app bundle is fetched once —
  self-hosted DuckDB-Wasm assets, not a runtime CDN — honoring §9's "local-only mode
  must work with zero network for OSS-style consumers."
- the donor monorepo's rollup endpoint gets a real, read-only consumer without any
  change to the donor monorepo's route or `walkWorktrees` core.

**Non-Goals:**

- A SQL scratchpad / ad hoc query surface. The donor consumer repo's dashboard offers one (D6
  page 7); canon's S9 surface is scoped to the five named panels only — an ad hoc
  query page is a candidate follow-up, not part of this change.
- Live collaboration or real-time updates beyond what the tiers give for free
  (design doc §11 non-goal).
- Replacing the donor monorepo's own dashboard — canon's dashboard is its own surface; the
  donor monorepo may choose to embed it later, but that is out of scope here (§11 non-goal).
- Write paths from the dashboard. Every panel query is read-only; there is no
  companion mutation endpoint (mirrors the donor monorepo's own `GET /api/openspec/changes` R3
  posture: author paths stay in the source tooling, not the surface).
- Generalizing the donor-rollup wiring to a plugin-style adapter framework for
  arbitrary consumer-repo dashboards. It is one optional, donor-shaped config field
  (`dashboardRollupUrl`) and one ingest step; a general adapter registry is S3/S12
  territory, not S9's.

## Decisions

### D1. `canon report` is a Rust crate (`canon-report`) driven off S2's DuckDB views, not a second aggregation layer

- **Decision:** `crates/canon-report` reads the `stg_/int_/mart_` views S2's
  `canon-store` already exposes (specdb.sql pattern) and adds S9-owned marts (trust
  matrix, session costs, role memory, flywheel funnel, review burn-down) as new
  views in the same layer — not a parallel query path. `canon report` renders those
  views to markdown; `canon report --snapshot <dir>` exports the same views to
  Parquet + `manifest.json`.
- **Why:** D16's entire point is that there is exactly one place aggregate state is
  computed. Adding a second Rust-side aggregation layer beside the DuckDB views
  would recreate the "two stores, no documented join key" failure S1 was built to
  kill (design doc §1).
- **Alternatives:** compute marts ad hoc in Rust structs bypassing DuckDB (rejected
  — duplicates SQL-expressible logic in a second language, doubles the surface that
  can drift); have the dashboard query the tiers directly at runtime instead of a
  snapshot (rejected in D3 below — a browser cannot reach git/pg/r2 tiers).

### D2. Report `--check` is regenerate-and-diff, never a heuristic freshness check

- **Decision:** `canon report --check` renders the report in memory from current
  inputs and compares it byte-for-byte against the existing `canon/REPORT.md` (or
  configured path). No existing file → exit 1 with a "generate first" message
  (parity.py's MISSING case). Byte match → exit 0, "no drift". Byte mismatch →
  exit 1, "DRIFT — regenerate with `canon report`" (the donor parity harness's
  `cmd_report`, lifted near-verbatim). The report header
  embeds ONLY input-derived digests (corpus/policy/ledger-head hash) — **no
  `source_git_sha` and no `generated_at` or other timestamp field**. A committed
  `canon/REPORT.md` can never contain the hash of the commit that adds it (that
  commit's hash is a function of the report's own bytes), so embedding `git
  rev-parse HEAD` here would make every committed report drift on the very next
  `--check`, permanently, the moment HEAD moves past the commit the report was
  rendered before (decision 11 extends: not just wall-clock timestamps, but any
  input a committed generated file cannot itself hold, is excluded from the
  drift-checked header). `source_git_sha` still exists — as `--snapshot`'s
  `manifest.json` field (D3 below), which is never drift-checked, so it can safely
  carry output-inclusive provenance the report header cannot.
- **Why:** matches D16's "if the report is committed, CI regenerates and fails on
  diff" contract exactly, and keeps the report reproducible from named inputs
  (attestation-style) rather than merely "looks current."
- **Alternatives:** hash-only comparison (rejected — a byte diff is what CI actually
  needs to show a reviewer; a hash mismatch alone doesn't explain what drifted);
  embedding `generated_at` for human readability (rejected — decision 11 is explicit
  that generated artifacts are timestamp-free; the digest header already answers
  "generated from what inputs").

### D3. Snapshot format is Parquet + a declared `manifest.json`, not the native store format or a live query path

- **Decision:** `canon report --snapshot <dir>` writes one `COPY … TO
  '<table>.parquet' (FORMAT parquet)` per exported mart (byte-identical filenames to
  table names, avoiding the `EXPORT DATABASE` escaping bug the donor parity harness's D2 hit with
  digit-containing table names) plus a single `manifest.json`:
  `{generated_at, source_git_sha, source_digest, tables: [{table, file}]}`. The
  dashboard never enumerates the snapshot directory — it reads `manifest.json` as
  the declared table→file map, exactly D2's `spec-status-dashboard` reasoning.
- **Why:** Parquet is DuckDB's version-stable cross-engine interchange; the CLI's
  native store format and a pinned DuckDB-Wasm build are not guaranteed compatible
  (storage-format skew, same root cause D2 documented). A snapshot (not a live
  connection) is required because a browser sandbox cannot read arbitrary tier
  files (git-tier paths, pg credentials, r2 buckets) — D2's "why a snapshot, not
  live" reasoning applies unchanged.
- **Alternatives:** ship the native store binary (rejected — storage-format skew);
  bundle raw source and rebuild marts in-browser (rejected in D2's donor design —
  ships the whole ledger to the browser and reimplements mart SQL in JS, exactly
  the "second aggregation layer" D1 rejects); a local API server the dashboard
  queries live (rejected — needs a running process, not static/deployable, and
  duplicates `canon report`'s own generation path).

### D4. Dashboard stack is native JS `@duckdb/duckdb-wasm`, self-hosted — not a CDN bootstrap

- **Decision:** `packages/dashboard` is a Bun/Vite + TypeScript static app using the
  `@duckdb/duckdb-wasm` npm package directly. Unlike the donor parity harness's Dart wrapper
  (`dart_duckdb` 1.4.4), whose web loader is hardcoded to `getJsDelivrBundles()`
  with no override (D3 of the donor design), the plain JS package supports manual
  bundle selection: the wasm/worker/eh files are downloaded once at build time (the
  documented manual-download path — duckdb.org's Wasm instantiation docs describe
  fetching the same files jsDelivr serves) and shipped under `packages/dashboard/
  dist` alongside the app, so first load needs no runtime CDN fetch.
- **Why:** design doc §9 requires "local-only mode (git tier alone) must work with
  zero network for OSS-style consumers." The donor parity harness's dev-only tool explicitly
  accepted a runtime CDN dependency as an MVP trade-off (deferred self-hosting);
  canon cannot make the same trade because zero-network local-only operation is a
  stated security/operational requirement, not a nice-to-have, and the JS package
  (unlike the Dart wrapper) makes self-hosting a documented, low-cost path rather
  than a package fork.
- **Alternatives:** accept the CDN dependency like the donor MVP (rejected — directly
  violates §9); fork/patch a Dart wrapper to self-host (moot — canon's stack is
  Bun/TS, not Flutter, so there is no wrapper to patch in the first place).

### D5. Panel marts are additive views in S2's layer; the donor monorepo's rollup endpoint is one optional ingest step, not a new adapter framework

- **Decision:** the five panels are backed by new `mart_trust_matrix`,
  `mart_session_costs`, `mart_role_memory`, `mart_flywheel_funnel`, and
  `mart_review_burndown` views added to S2's DuckDB-view layer, each reading S1
  join-spine tables (`Change`/`Task`/`EvidenceRecord` for the trust matrix,
  `Session`/`Run` keyed by `session_id` for costs, `StrategyItem` for role memory,
  the S4 verdict stream + S6/S7/S8 events for the funnel and burn-down). The donor monorepo's `GET
  /api/openspec/changes` is wired as one conditional ingest step: when `canon.yaml`
  sets `dashboardRollupUrl`, `canon report`'s ingest phase calls the endpoint with
  the required `X-User-Id` header and folds the returned `RolledChange[]` rows
  (`{worktree, branch, slug, route, created, proposalTitle}`, unchanged shape) into
  `mart_trust_matrix` as supplemental rows tagged by their source (`donor-rollup`)
  alongside canon's own git-tier change scan (S1/S4), so an operator sees in-flight
  changes across every worktree even when running the dashboard from one checkout.
- **Why:** the endpoint's whole value is a **cross-worktree** view canon's own
  single-checkout git-tier scan cannot produce cheaply; wiring it as a data source
  (not reimplementing worktree-walking in Rust) reuses the existing, tested
  `walkWorktrees()` logic the donor monorepo already ships instead of duplicating it.
- **Alternatives:** reimplement `git worktree list --porcelain` walking in
  `canon-ingest` (rejected — duplicates tested logic the donor monorepo already has for no
  benefit; canon's own git-tier scan already covers the single-checkout case S4
  needs); make the donor-rollup adapter generic/pluggable for any consumer repo
  (rejected as non-goal — no other consumer repo exposes an equivalent endpoint
  today; a generic registry is premature abstraction until a second consumer needs
  one).

## Risks / Trade-offs

- **[Risk] A stale snapshot silently under-represents current state.** →
  **Mitigation:** the freshness banner always renders `manifest.json`'s
  `source_git_sha`/`source_digest`/`generated_at`; the dashboard's README and
  companion skill both state explicitly that "matches `canon report`" is scoped to
  the recorded snapshot input, not the live checkout (D3, mirroring the donor's D7
  consequence).
- **[Risk] Self-hosting the Wasm bundle (D4) adds a build-time asset-fetch step and
  a version-pin maintenance burden `@duckdb/duckdb-wasm` upgrades will require.** →
  **Mitigation:** pin the exact package version in `packages/dashboard/package.json`
  (same discipline as the donor's pinned `1.29.1-dev222.0`); the bundle-fetch step
  runs once at `packages/dashboard` build time (CI-cacheable), not per dashboard
  load.
- **[Risk] The donor-rollup data source (D5) is one more moving part specific to one
  consumer repo, and its `X-User-Id` auth model assumes the donor monorepo's own gateway
  populates that header — calling it outside the donor monorepo's gateway context will 401.** →
  **Mitigation:** the field is optional and donor-only; `canon report`'s ingest step
  treats a 401/503 from this source as fail-soft (log + skip, exactly S8's
  "retrieval is advisory" fail-soft posture) — the trust-matrix panel still renders
  from canon's own git-tier scan with the donor-rollup rows simply absent.
- **[Risk] `EXPORT DATABASE`-style filename escaping (digits in table names) silently
  breaks the manifest's table→file map.** → **Mitigation:** D3 uses one explicit
  `COPY "<table>" TO '<table>.parquet'` statement per table (not `EXPORT DATABASE`),
  the exact fix the donor design verified against the digit-escaping bug.

## Migration Plan

Net-new surface; no prior `canon report` or dashboard exists to migrate from. Roll-
out order: `crates/canon-report` (markdown + `--check`) lands first and is
independently useful (CI can gate on drift immediately); `--snapshot` export and
`packages/dashboard` land second, consuming the same views. No consumer repo config
changes are required except the donor monorepo opting in to `dashboardRollupUrl` (additive,
optional field on `canon.yaml`).
