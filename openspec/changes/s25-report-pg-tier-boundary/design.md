# Design — s25 report pg-tier boundary

## Current state (accurate baseline, verified)

- **`crates/canon-store/sql/views.sql`'s `stg_records` (line 83) is `stg_git_records UNION ALL stg_r2_records` — no `pg` source anywhere in the file.** Every `mart_*` a `canon-report` panel reads is built on `stg_records` (directly or via `int_*`), so a kind routed to `pg` contributes zero rows to any panel, unconditionally, regardless of whether a live `pg` database is reachable at report time.
- **`canon.yaml`'s `TierPolicy.routing` (`crates/canon-store/src/policy.rs`) is the single source of truth for which tier a kind resolves to.** `TierPolicy::from_yaml` is a PURE YAML parse — no I/O beyond reading the file, no socket, no live-tier connection — already the exact parse `canon_cli::report::resolve_roots` performs today for `tiers.git.root` (`crates/canon-cli/src/report.rs:56-72`), and the exact parse `canon_cli::tiers::build_tiers`/`build_lenient_tiers_for_kind` perform for `canon query`'s own tier fan-out (s22 `query-tier-degradation`).
- **`canon-report` already depends on `canon-store`** (`crates/canon-report/Cargo.toml:19`) — for `RecordKind` (`crate::digest`) and, after this change, `TierPolicy`/`TierKind` (both pure config types, `crates/canon-store/src/policy.rs`, zero I/O beyond the YAML parse itself). No NEW crate dependency is introduced anywhere by this change.
- **`canon-report`'s `report()` (`crates/canon-report/src/lib.rs`) already reads a file directly off `inputs.repo_root`** — `digest::DigestHeader::compute` reads `<repo_root>/canon/policy.yaml` (a DIFFERENT file, `crate::digest::POLICY_YAML_RELATIVE_PATH`) with the identical "absent/malformed → an honest placeholder, never a panic or an `Err`" posture this change's `pg_routed_kinds` also adopts for `<repo_root>/canon.yaml`. This is an established pattern in this crate, not a new one.
- **`canon report`'s stderr today prints exactly one line per invocation** (`crates/canon-cli/src/main.rs::run_report`): `"canon report: wrote …"` / `outcome.message(&report_path)` (`--check`) / `"canon report --snapshot: wrote … table(s) to …"`. There is no precedent for a SECOND, warning-shaped stderr line — this change is the first.

## Decisions

- **D1 — Config-derived (static `canon.yaml` `routing`), never a live `pg` row count or connection.** Two shapes were weighed: (a) `canon report` opens a live `pg` connection (reusing `PgTier::connect`) and counts rows per `pg`-routed kind, embedding the counts in the note; (b) compute the routed-KIND SET alone from `TierPolicy::routing`, with no live read at all. **(b) is accepted.** (a) breaks the report's foundational offline/deterministic/drift-checkable contract (S9 decision 11: `canon report --check` is a byte-diff gate over the exact same rendering logic run twice — a live row count would make the report's content depend on `pg`'s current state, so `--check` could report drift with NO `canon.yaml`/corpus change at all, or worse, silently differ between a CI run with no DSN and an operator's local run with one). (b) is exactly as deterministic as every other section of the report: two runs at unchanged `canon.yaml` produce byte-identical output, with or without a live `CANON_PG_DSN` even set — proven by this change's own `two_renders_of_an_unchanged_multi_tier_config_are_byte_identical`-shaped acceptance test. The operator-locked scope (proposal.md "Why") requires this reading directly.
- **D2 — Note placement + exact deterministic format.** The note is a NEW top-level `## Tiers not reflected` markdown section, placed immediately after the existing `## Inputs (digest)` block and before `## Trust matrix` — never inserted mid-panel, never replacing an existing panel's own empty-row placeholder. Rendered ONLY when the routed-kind set is non-empty (an empty `Vec` renders NOTHING — no heading, no blank line — so every existing git+r2-only fixture's rendered output is byte-for-byte IDENTICAL to what it rendered before this change existed). Content, in order: a fixed heading, one blockquote sentence naming `git`+`r2` as the marts' only sources and `canon query --kind <kind>` as the escape hatch, then one `- \`<kind>\`` bullet per routed kind, SORTED ascending by `RecordKind::as_str()` (never `HashMap` iteration order — the exact same "sorted, not hash-order" discipline `crate::digest::canonical_blob` already applies to record digests). The CLI's stderr `WARN` line is a single-line, comma-joined rendering of the SAME sorted kind list, prefixed `canon report: WARN ` (matching every existing `canon report[…]: ` stderr-line convention in `main.rs::run_report`) — computed via the identical `pg_routed_kinds` call the markdown note uses, so the two representations can never name a different kind set or a different order.
- **D3 — `canon-report` reads `canon.yaml` itself; `canon-cli` does NOT thread a `pg_routed_kinds` field through `ReportInputs`.** Two shapes were weighed: (i) `canon_cli::report::resolve_inputs` parses `canon.yaml` (it already does, for `tiers.git.root`) and passes a new `pg_routed_kinds: Vec<RecordKind>` field on `ReportInputs` into `canon-report`; (ii) `canon-report`'s own `report()` reads `<repo_root>/canon.yaml` directly (a NEW `tier_boundary::pg_routed_kinds(&inputs.repo_root)` call, mirroring `digest::DigestHeader::compute`'s existing direct read of `<repo_root>/canon/policy.yaml`), with no `ReportInputs` field change at all. **(ii) is accepted.** Neither shape adds a live-`pg`/`canon-store`-live dependency to `canon-report` (`TierPolicy::from_yaml` is a pure parse either way, and `canon-store` is already a `canon-report` dependency) — the deciding factor is API stability and precedent: (i) would change `ReportInputs`'s public shape (a breaking change for `crates/canon-report/src/bin/canon-report.rs` and any future non-`canon-cli` caller) and duplicate config-reading logic that already lives in `canon-cli::report::resolve_roots` for a DIFFERENT purpose (`tiers.git.root`, not `routing`); (ii) adds a self-contained module reading a file `canon-report` already has direct, established precedent for reading off `repo_root` (`digest`'s own `POLICY_YAML_RELATIVE_PATH`), with zero signature changes to `ReportInputs`/`report()`/`write_report()`/`check_report()`. `canon-cli`'s `run_report` calls the SAME public `canon_report::tier_boundary::{pg_routed_kinds, warn_line}` functions independently (for the stderr line), guaranteeing the CLI-side warning and the library-side markdown note are always computed from the identical function over the identical repo root — never two independently-maintained derivations that could drift apart.
- **D4 — Non-goal: no `stg_pg_records`, no live `pg` read, data stays in `canon query`.** `crates/canon-store/sql/views.sql` gains ONLY a doc-comment addition at `stg_records` (naming the boundary, matching the file's own "name every stub/proxy/gap" convention already established at `int_evidence_verdicts`) — no `CREATE VIEW`/`CREATE OR REPLACE VIEW` statement changes anywhere in the file. A `pg`-routed kind remains reachable EXCLUSIVELY through `canon query --kind <kind>` (`canon-cli`'s own live tier fan-out, `canon_store::registry::TierRegistry::query`, s22 `query-tier-degradation` — already correct and live-proven in the round-3 dogfood). This is the operator-locked scope from proposal.md "Why": the fuller fix (a `pg`→Parquet snapshot export feeding a real `stg_pg_records`, floated as the round-3 review's own "if the fuller fix is in scope" alternative) is explicitly OUT of scope for this change.

## Risks

- **R1 — A repo whose `canon.yaml` is present but genuinely malformed (fails `TierPolicy::from_yaml`) degrades to the SAME empty-set/no-note behavior as a repo with no `canon.yaml` at all.** Accepted: `canon fmt`/`canon gate check` are the blessed surfaces for a malformed `canon.yaml` to surface loudly (existing behavior, untouched by this change); `canon report`'s boundary note is deliberately fail-soft here so a config error elsewhere in `canon.yaml` (e.g. a malformed `aging` duration, unrelated to `routing`) can never turn an otherwise-successful `canon report` run into a hard failure. This exactly mirrors `crate::digest::DigestHeader::compute`'s existing precedent for `canon/policy.yaml`.
- **R2 — A future `RecordKind` added to the closed 12-kind set without a corresponding `mart_*`/`stg_*` change could silently start appearing in `## Tiers not reflected` even when it is actually GIT/R2-routed, if `views.sql` and `canon.yaml`'s `routing` table ever disagree about which kinds a mart covers.** Not possible today: `pg_routed_kinds` reads `canon.yaml`'s `routing` directly (the same source of truth `TierRegistry::tier_for` uses for every live read/write), and `stg_records`'s own `UNION ALL` is exhaustively `git ∪ r2` with no third source — so "routed to `pg`" and "invisible to the marts" are definitionally the same set, not two independently-maintained lists that could drift. Documented at `tier_boundary.rs`'s own module doc as the invariant this change relies on.
- **R3 — `SNAPSHOT_TABLES`/`canon report --snapshot`'s Parquet export set is untouched by this change** (D2: the note is markdown-only, not a DuckDB view/mart) — a `--snapshot`-only consumer (e.g. `packages/dashboard`) gets no boundary signal at all. Accepted as an explicit non-goal (proposal.md): the manifest/Parquet contract is a separate, already-shipped surface (S9 part2/part3) this change does not extend; the blessed surface for the boundary note is the markdown `canon report` (default) invocation, matching the round-3 finding's own framing ("the report" = the markdown output a daily-driver actually reads).

## Sequencing

- **P1 — `crates/canon-report/src/tier_boundary.rs` (new module): `pg_routed_kinds`, `render_note`, `warn_line` (D1-D3).** Standalone; depends only on `canon_store::policy::TierPolicy`/`TierKind` and `canon_model::envelope::RecordKind`, both already-shipped types.
- **P2 — `crates/canon-report/src/render.rs`: `render()` gains the `pg_routed_kinds: &[RecordKind]` parameter and the `## Tiers not reflected` block (D2), after P1.**
- **P3 — `crates/canon-report/src/lib.rs::report()`: computes `pg_routed_kinds` from `inputs.repo_root` and threads it into `render::render` (D3), after P1/P2.**
- **P4 — `crates/canon-cli/src/main.rs::run_report`: the stderr `WARN` block, calling `tier_boundary::{pg_routed_kinds, warn_line}` directly (D3), independent of P2/P3 — needs only P1's public functions to exist.**
- **P5 — `crates/canon-store/sql/views.sql`: `stg_records`'s doc-comment addition (D4), independent of P1-P4 — a pure documentation change with no code dependency either direction.**
- **P6 — Tests, after P1-P4.** `crates/canon-report/tests/tier_boundary.rs` (multi-tier note rendering, git-only no-note, byte-identity across repeated runs); `crates/canon-cli/tests/report_tier_boundary.rs` (stderr WARN present/absent). Every EXISTING test in both crates is re-run unmodified as a regression guard (proposal.md Impact: none of their fixtures write a `canon.yaml`, so the new code path returns the empty set for all of them).
- **P7 — Closure.** `bunx openspec validate --strict s25-report-pg-tier-boundary`; `cargo build --workspace` + `cargo clippy --workspace --all-targets -- -D warnings` + `cargo test --workspace --no-fail-fast` green; `canon gate check` byte-identity re-confirmed (no `canon-gate` source file touched — `crates/canon-report/tests/gate_independence.rs`'s existing assertions are the direct regression proof, unmodified).
