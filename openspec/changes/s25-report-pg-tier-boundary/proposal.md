## Why

The loom round-3 dogfood — canon's first LIVE run against a real multi-tier (git+pg+r2) repo — surfaced a unanimous, source-triangulated finding, F2 (`target/usage-review/loom/SYNTHESIS-ROUND3.md`, "Consensus finding — F2"):

- **`canon report` is structurally blind to the `pg` (hot) tier.** All four review personas independently read `crates/canon-store/sql/views.sql` and confirmed: `stg_records = stg_git_records ∪ stg_r2_records` — there is no `stg_pg_records`, and zero occurrences of `pg`/`postgres` anywhere in the file. Every mart (`mart_trust_matrix`, `mart_scope_status`, …) builds on `stg_records`, so in loom's config (`task`/`session`/`run`/`event`/`handoff` → `pg`) all six report panels render `_No rows._` while `canon query --kind task` returns 5 real rows from the same `pg` database — confirmed with a live DSN set and data present, so this is structural, not an env artifact.
- **The failure is SILENT.** No warning, no error, no non-zero exit, no digest mismatch (`crates/canon-report/src/render.rs`'s digest header only hashes `change`/`task`/`scenario`/`evidence_record`/`review`/`divergence` off the GIT tier — `crates/canon-report/src/digest.rs`). Every persona called this *worse* than a hard failure: the report carries false authority ("nothing tracked here") for a daily-driver who trusts it.
- **The affected kinds carry no aging rule in loom's config**, so the data is *permanently* invisible to the report, not merely "hasn't aged to a durable tier yet." Plan-agent (architecture judge) ruled this a genuine seam: `views.sql`'s own comment convention is to name every stub/proxy/gap it contains (`int_evidence_verdicts`'s explicit STUB note is the precedent) — `pg`'s silent absence from `stg_records` is the one undocumented boundary in the file.

**Operator-locked scope for this change** (session decision, not re-litigated here): do NOT make `canon report` read a live `pg` connection — that would break the report's foundational offline/deterministic/drift-checkable contract (S9 decision 11, `canon report --check`'s byte-diff gate). Instead, kill the SILENT failure: emit a LOUD, CONFIG-DERIVED, deterministic boundary note + a stderr warning, and leave the `pg`-routed data reachable exactly where it already is — `canon query --kind <kind>` (unaffected, already correct per s22 `query-tier-degradation`).

## What Changes

- **`canon-report` computes, from `canon.yaml`'s STATIC `TierPolicy.routing` table (never a live `pg` row count or connection), the set of `RecordKind`s routed to `pg`.** A new `crates/canon-report/src/tier_boundary.rs` module: `pg_routed_kinds(repo_root: &Path) -> Vec<RecordKind>` reads `<repo_root>/canon.yaml` and parses it with the SAME `canon_store::policy::TierPolicy::from_yaml` the CLI's own `canon_cli::report::resolve_roots` already calls for `tiers.git.root` — a pure YAML parse, no socket, no new dependency (`canon-report` already depends on `canon-store`). Fail-soft to an empty `Vec` (never a panic/`Err`) when `canon.yaml` is absent or unparseable, mirroring `crate::digest::DigestHeader::compute`'s existing "no `canon/policy.yaml` → placeholder, never an error" posture for a sibling repo-root config file. Sorted ascending by `RecordKind::as_str()` for deterministic output regardless of `HashMap` iteration order.
- **When that set is non-empty, `canon report` renders a `## Tiers not reflected` section** in `REPORT.md`, placed right after the `## Inputs (digest)` block and before `## Trust matrix` — naming every `pg`-routed kind and pointing at `canon query --kind <kind>`. Timestamp-free, sorted, byte-stable across two runs at unchanged `canon.yaml` (so `canon report --check` stays green on unchanged config); entirely ABSENT (no section at all) when the set is empty, so every existing git+r2-only fixture/test renders byte-identically to before this change.
- **`canon report` (the CLI) also emits a one-line stderr `WARN`** naming the same config-derived kind set — computed via the SAME `pg_routed_kinds` call, so the markdown note and the stderr line can never disagree. Silent (no `WARN` line) for a repo with nothing routed to `pg`.
- **`crates/canon-store/sql/views.sql` gains a doc comment at `stg_records`** stating explicitly that the `pg` tier is intentionally NOT staged there (report = git+r2 durable-tier view only; `pg` stays reachable via `canon query`), matching the file's own established convention of naming every stub/proxy/gap it contains (`int_evidence_verdicts`'s STUB note is the direct precedent).
- **No change to `mart_*`/`stg_*` SQL semantics, `canon-report`'s Parquet snapshot export set, or any panel's row content.** This change adds ONE new, purely additive markdown section plus a CLI stderr line — every existing panel's rendered output is byte-for-byte unchanged for any fixture/repo where nothing routes to `pg`.

### Added Capabilities

- `report-pg-tier-boundary`: `canon report`'s config-derived, deterministic boundary surface for the `pg` (hot) tier — a `## Tiers not reflected` markdown section plus a stderr `WARN`, both computed from `canon.yaml`'s static `routing` table (never a live `pg` read), naming every `RecordKind` the report's marts cannot see and pointing the operator at `canon query --kind <kind>` instead.

### Explicit non-goals

- **NO live `pg` read anywhere in `canon-report` or `canon-store`'s DuckDB view layer.** No `stg_pg_records`, no `read_parquet`/`read_text` over a `pg` export, no new live-database dependency added to `canon-report`. The report's offline/deterministic/drift-checkable contract (S9 decision 11) is preserved exactly as-is — this change is a documentation/warning surface over EXISTING config, not a new data path.
- **No change to `canon query`'s existing `pg`-tier read path.** `canon query --kind <kind>` for a `pg`-routed kind already works correctly (s22 `query-tier-degradation`, live-proven in the round-3 dogfood) — this change does not touch `crates/canon-cli/src/query.rs`, `canon-store`'s `PgTier`, or `TierRegistry`.
- **No new `RecordKind`, no change to the closed 12-kind set.** `RecordKind::ALL.len() == 12` and its assertion sites are untouched.
- **No change to `canon gate check`'s inputs or verdicts.** `canon-gate` reads nothing from `canon-report`; this change touches no `canon-gate` source file. `canon gate check` verdicts are byte-identical before and after, for any corpus — connector-never-authority is preserved (the `pg`-boundary note is read-only reporting, exactly like every other `canon report` panel).
- **No CLI flag changes.** `canon report [--repo] [--check] [--snapshot <dir>]` keeps its existing argument shape; the new section/warning appears automatically, with no opt-in/opt-out flag, matching every existing panel's non-optional inclusion.
- **No `canon report --snapshot`/`manifest.json` change.** The boundary note is a markdown-only addition (config-derived text, not a DuckDB view/mart); `SNAPSHOT_TABLES`'s Parquet export set is untouched.

## Impact

- **`canon-report`**: new `src/tier_boundary.rs` (`pg_routed_kinds`, `render_note`, `warn_line`), `src/lib.rs` (`report()` computes `pg_routed_kinds` from `inputs.repo_root` and threads it into `render::render`), `src/render.rs` (`render()` gains a `pg_routed_kinds: &[RecordKind]` parameter; embeds the `## Tiers not reflected` section when non-empty).
- **`canon-report` tests**: new `tests/tier_boundary.rs` (multi-tier boundary-note rendering, git-only/no-`canon.yaml` no-section case, byte-identity across two runs at unchanged config). Every existing `canon-report` test (`byte_stability.rs`, `fresh_repo.rs`, `marts.rs`, `snapshot.rs`, …) is unaffected — none of their fixture repos write a `canon.yaml`, so `pg_routed_kinds` returns empty and the new section never renders for them.
- **`canon-cli`**: `src/main.rs::run_report` gains one stderr-`WARN` block, computed via `canon_report::tier_boundary::{pg_routed_kinds, warn_line}` right after `--repo` resolution, before dispatching to `--snapshot`/`--check`/the flagless write. No change to `crates/canon-cli/src/report.rs`'s `--repo`-resolution logic itself.
- **`canon-cli` tests**: new `tests/report_tier_boundary.rs` (stderr WARN on a multi-tier repo naming the routed kinds; no WARN on a git-only repo). Existing `tests/report.rs` is unaffected (its fixture repos carry no `canon.yaml`).
- **`crates/canon-store/sql/views.sql`**: `stg_records`'s doc comment grows to name the `pg` boundary explicitly. No `CREATE VIEW` statement changes; every existing `canon-store`/`canon-report` SQL-layer test is unaffected.
- **No new crate.** `canon-model`/`canon-ingest`/`canon-vocab`/`canon-plugin`/`canon-learn`/`canon-gate`/`canon-store`'s Rust source (beyond the SQL doc comment) is unchanged.
- **`packages/dashboard`**: unaffected — it consumes `manifest.json`/Parquet snapshots only, never `REPORT.md`'s markdown text.
