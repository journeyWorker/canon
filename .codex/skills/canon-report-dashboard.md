# canon-report-dashboard

> How to run canon report (the generated-never-edited markdown status report, its --check drift gate, and its --snapshot Parquet export) and canon dashboard (the zero-network web app that renders the same numbers from a snapshot). Use when you need the current trust-matrix/session-costs/role-memory/flywheel/review-burndown status of a canon-managed repo, when CI needs to gate on report drift, or when a snapshot needs regenerating for the dashboard.

# canon-report-dashboard

canon's single status surface: `canon report` renders five panels to
markdown, and `canon dashboard` serves a small zero-network web app that
renders the SAME numbers from a Parquet snapshot. Neither computes
aggregates itself â€” both read canon's precomputed views, so they can
never disagree.

## Generated, never hand-edited

`.canon/REPORT.md` carries a leading `<!-- GENERATED by \`canon report\` â€¦
DO NOT EDIT. -->` comment. If it looks stale or wrong, the fix is
**always** re-running `canon report` (or fixing the underlying
ledger/policy it renders) â€” never hand-editing the file. The report
header embeds only input-derived digests (corpus hash, policy hash,
ledger-head hash) and **no timestamp, no `source_git_sha`**, so a
committed report is reproducible from named inputs and `canon report
--check` can byte-diff it deterministically. `source_git_sha` belongs
only in `--snapshot`'s `manifest.json` (below), which is never
drift-checked.

## `canon report [--repo <dir>]`

Renders the current five-panel report to `<repo>/.canon/REPORT.md`.
`--repo` resolves through the nearest-`canon.yaml`-ancestor walk every
subcommand uses â€” omit it (or pass `.`) to run from any subdirectory.

```bash
canon report                 # writes .canon/REPORT.md
canon report --repo /path/to/repo
```

## `canon report --check` â€” the CI drift gate

Regenerates the report in memory and byte-compares it against the
committed `.canon/REPORT.md`:

- `MISSING` (exit 1) â€” no report has ever been written; run `canon report` first.
- no drift (exit 0) â€” the committed file is byte-identical to what the
  current inputs would render now.
- `DRIFT` (exit 1) â€” the committed file disagrees with current inputs
  (hand-edited, or the corpus/policy/ledger moved and nobody re-ran
  `canon report`).

```bash
canon report --check   # wire into CI exactly like canon fmt --check / canon gate check
```

Wire it as a CI step or pre-push hook the same way `canon gate
install-hooks` wires `canon gate check`; run `canon report --check`
directly wherever CI already runs those.

## `canon report --snapshot <dir>`

Exports the same five views to `<dir>/<table>.parquet` plus a
`<dir>/manifest.json`:

```json
{
  "generated_at": "...",
  "source_git_sha": "...",
  "source_digest": "...",
  "tables": [
    { "table": "mart_trust_matrix", "file": "mart_trust_matrix.parquet" },
    { "table": "mart_session_costs", "file": "mart_session_costs.parquet" },
    { "table": "mart_role_memory", "file": "mart_role_memory.parquet" },
    { "table": "mart_flywheel_funnel", "file": "mart_flywheel_funnel.parquet" },
    { "table": "mart_review_burndown", "file": "mart_review_burndown.parquet" }
  ]
}
```

This is where `source_git_sha`/`generated_at` live (never in the
drift-checked markdown header) â€” a snapshot is output-inclusive
provenance.

```bash
canon report --snapshot /tmp/canon-snap
```

## `canon dashboard [--repo <dir>] [--snapshot <dir>] [--port <n>]`

Serves the dashboard web app locally, pointed at a real snapshot:

```bash
# build the app once
bun install && bun run --cwd packages/dashboard build

canon dashboard                       # regenerates a fresh snapshot every run, serves it
canon dashboard --snapshot /tmp/canon-snap --port 8080
```

- `--snapshot` omitted: `canon dashboard` calls `canon report --snapshot`
  itself every run, into the conventional `.canon/dashboard-snapshot`
  scratch directory â€” always fresh.
- `--snapshot <dir>` given: served AS-IS when `<dir>` already has a
  `manifest.json` (a separately generated snapshot, or a downloaded CI
  artifact) â€” never silently regenerated over it. If it has no
  `manifest.json`, one is generated there once.
- Prints the URL to open, then serves until Ctrl-C. If the app is not
  built, `canon dashboard` fails with the exact build command to run.

The dashboard's freshness banner shows the snapshot's
`generated_at`/`source_git_sha`/`source_digest` with a "scoped to this
snapshot, not necessarily the live checkout" note â€” treat "matches
`canon report`" as "matches the inputs that produced THIS snapshot".

## Reading the five panels

- **Trust matrix** (`mart_trust_matrix`) â€” change/task Ă— `covered` Ă—
  `green` Ă— `who`. `covered` = required evidence exists per policy;
  `green` = the latest verdict actually passed.
- **Session costs** (`mart_session_costs`) â€” token/cost by
  role/repo/session, `session_id`-keyed.
- **Role memory** (`mart_role_memory`) â€” strategies, hit rate, and an
  effect proxy per role namespace (see `canon-reward`).
- **Flywheel funnel** (`mart_flywheel_funnel`) â€” verdicts â†’ distilled â†’
  retrieved â†’ applied counts. A large drop between two stages is the
  actionable signal.
- **Review burn-down** (`mart_review_burndown`) â€” per-day
  evidence/divergence counts plus a running "divergences still open"
  total. A running total that never trends down means feedback isn't
  being closed out.

## What this skill does NOT cover

- The underlying view layer that computes the marts â€” see `canon-storage`.
- `canon gate`'s evidence-gated checkbox flip / trust ladder â€” see
  `canon-gate`. The trust-matrix panel READS the same corpus `canon
  gate check` validates but never gates or flips anything.
- Strategy distillation/promotion â€” see `canon-reward`; the role-memory
  panel is a read-only view over that store.
