<span class="canon-kicker">Why canon</span>

## Evidence-gated completion, not self-reported completion

<CardGrid stagger>
  <Card title="The problem" icon="warning">
    Agents claim done. Repositories cannot verify what "done" means.
  </Card>
  <Card title="The core: evidence-gated completion" icon="approve-check">
    `canon gate` only flips a checkbox when policy-required evidence exists,
    carries a pass verdict, and hasn't gone stale — with stable failure
    classes when it doesn't, and a real exit code either way.
  </Card>
</CardGrid>

<span class="canon-kicker">What the same evidence extends into</span>

<CardGrid stagger>
  <Card title="Systematic spec planning" icon="pen">
    Specs are a first-class corpus — structured change lifecycles,
    Gherkin scenario ledgers, and a format authority (`canon fmt`) that keeps
    every artifact valid, versioned, and machine-readable.
  </Card>
  <Card title="Unified session logging" icon="bars">
    One join spine across agent CLIs. `canon ingest` normalizes transcripts
    from Claude Code, Codex, and friends into tiered storage — git, hot, and
    cold — with a single documented join key.
  </Card>
  <Card title="Role-scoped strategy memory" icon="puzzle">
    Reviews and verdicts distill into per-role strategies — planning, design,
    dev, test, review — each voice evolving on the same shared backbone.
    `canon retrieve` surfaces them before the next run starts.
  </Card>
</CardGrid>

<span class="canon-kicker">Honest about limits</span>

## What canon proves — and what it doesn't

Canon strongly verifies:

- that policy-required evidence exists
- that evidence is recorded as pass or fail
- who recorded it
- that the target code hasn't changed out from under the evidence (freshness)
- that a completion mark passed a defined procedure

Canon does **not** prove:

- whether a test actually verifies the requirement
- whether a reviewer judged correctly
- that a malicious user fabricated plausible-looking evidence
- mathematical or semantic correctness of the program

**Canon verifies the structure, provenance, verdict, and freshness of
evidence. It does not decide whether a test or reviewer is semantically
correct.**

That's the boundary on purpose: `coverage` — does evidence exist — is
tracked separately from `verdict` — did it pass. Tools that oversell
themselves blur that line; canon draws it deliberately.

<span class="canon-kicker">Quick start</span>

## See it in 60 seconds

```bash
mkdir canon-demo && cd canon-demo
bunx @journeykit/canon demo init      # scaffolds a throwaway evidence loop
bunx @journeykit/canon gate check     # RED
# canon gate check: 1 violation(s)
#
# uncovered-cell (1):
#   uncovered-cell auth.login.01 — policy-required cell 'reviewer' has no matching-role evidence record
bunx @journeykit/canon demo attest    # records the reviewer's evidence
bunx @journeykit/canon gate check     # GREEN
# canon gate check: clean (0 violations)
```

Watch the gate flip from red to green the instant reviewer evidence lands —
no scenarios, review records, or regime keys required to see it happen.

## In any repo

```bash
# one Rust core, distributed through npm
bunx @journeykit/canon report          # where does this repo stand?
bunx @journeykit/canon gate check      # is "done" actually done?
bunx @journeykit/canon ingest sessions # fold agent transcripts into the ledger
```

canon is named for the musical form: one subject, taken up by many voices,
each developing it in its own register. [Read the concept →](/concepts/canon/)
