## 자가 보고가 아닌, evidence-gated 완료

<CardGrid stagger>
  <Card title="문제" icon="warning">
    에이전트는 완료라고 주장합니다. 레포지토리는 그 "완료"가 무엇을
    의미하는지 검증할 방법이 없습니다.
  </Card>
  <Card title="핵심: evidence-gated 완료" icon="approve-check">
    `canon gate`는 정책상 요구되는 증거가 존재하고, 통과 판정을 받았고,
    stale하지 않을 때만 체크박스를 넘깁니다 — 그렇지 않으면 안정적인
    실패 클래스로 알려주고, 어느 쪽이든 실질적인 종료 코드를 남깁니다.
  </Card>
</CardGrid>

## canon이 증명하는 것 — 그리고 증명하지 않는 것

canon이 강하게 검증하는 것:

- 정책상 요구되는 증거가 존재하는지
- 증거가 통과 또는 실패로 기록되었는지
- 누가 그것을 기록했는지
- 대상 코드가 증거 아래에서 바뀌지 않았는지(freshness)
- 완료 표시가 정의된 절차를 통과했는지

canon이 증명하지 **않는** 것:

- 테스트가 요구사항을 실제로 검증하는지
- 리뷰어가 올바르게 판단했는지
- 악의적인 사용자가 그럴듯해 보이는 증거를 조작하지 않았는지
- 프로그램의 수학적·의미적 정확성

**canon은 증거의 구조, provenance, 판정, freshness를 검증합니다. 테스트나
리뷰어가 의미적으로 옳은지는 판단하지 않습니다.**

바로 그 경계가 의도적입니다: `coverage` — 증거가 존재하는가 — 는
`verdict` — 통과했는가 — 와 분리되어 추적됩니다. 스스로를 과장하는
도구들은 이 경계를 흐리지만, canon은 이를 의도적으로 갈라놓습니다.

## 60초 만에 확인하기

```bash
mkdir canon-demo && cd canon-demo
bunx @journeykit/canon demo init      # scaffolds a throwaway evidence loop
bunx @journeykit/canon gate check     # RED
# canon gate check: 1 violation(s)
#
# uncovered-cell (1):
#   uncovered-cell auth.login.01 — policy-required cell 'reviewer' has no matching-role evidence record
bunx @journeykit/canon demo attest    # records the reviewer's evidence
bunx @journeykit/canon gate check     # GREEN
# canon gate check: clean (0 violations)
```

리뷰어 증거가 도착하는 순간 게이트가 red에서 green으로 바뀌는 것을
확인하세요 — 시나리오, 리뷰 레코드, regime key가 무엇인지 몰라도
이 흐름을 볼 수 있습니다.

## 어느 레포에서든

```bash
# 하나의 Rust 코어, npm으로 배포
bunx @journeykit/canon report          # 이 레포는 지금 어디까지 왔나?
bunx @journeykit/canon gate check      # "done"이 진짜 done인가?
bunx @journeykit/canon ingest sessions # 에이전트 트랜스크립트를 레저로
```

canon이라는 이름은 음악의 캐논에서 왔습니다: 하나의 주제를 여러 성부가
이어받아, 각자의 성역에서 발전시키는 형식.
