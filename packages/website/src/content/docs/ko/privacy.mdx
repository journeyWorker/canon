`canon ingest sessions`(S3 파이프라인)는 각 에이전트 CLI 자체의 로컬 transcript
파일을 canon-model `Session`/`Run`/`Event` record로 변환합니다. 이 페이지는
정확히 무엇을 읽는지, 무엇을 저장하는지, 그 데이터가 어디에 있는지, 그리고
redaction과 삭제 기능이 (아직) 어디까지 제공되는지를 있는 그대로 설명합니다.

## Canon이 읽는 것

`canon-ingest`는 네 개의 adapter를 제공하며, 각각 **로컬 transcript 파일만**
읽습니다 — 네트워크 호출도, 서드파티 API 호출도 없습니다:

- **omp/pi** — `~/.omp/agent/sessions/<encoded-cwd>/*.jsonl`와
  `~/.pi/agent/sessions/<encoded-cwd>/*.jsonl` (badlogic/pi-mono와 그 Oh My
  Pi fork는 동일한 JSONL 포맷을 공유하며, 두 홈 디렉터리를 모두 스캔합니다).
- **Claude Code** — `~/.claude/projects/<sanitized-cwd>/*.jsonl`.
- **Codex** — `${CODEX_HOME:-~/.codex}/sessions/*.jsonl`과
  `archived_sessions/*.jsonl`을 합집합으로 스캔합니다 (Codex CLI가 오래된
  세션을 rotate하는 위치입니다).
- **Hermes Agent** — `~/.hermes/state.db` SQLite 데이터베이스와,
  `~/.hermes/profiles/<name>/state.db` 아래의 프로필 데이터베이스들.

각 adapter는 환경 변수 override(`CANON_INGEST_OMP_SESSIONS_DIR`,
`CANON_INGEST_CLAUDE_SESSIONS_DIR`, `CODEX_HOME`, `HERMES_HOME`)와
`canon.yaml`의 `ingest.sources.<id>.roots` override도 지원합니다. 내 컴퓨터에
존재하지 않는 source root는 조용히 0건으로 스킵됩니다 — 에러가 아닙니다.

## Canon이 저장하는 것

한 번의 pass는 스캔된 모든 transcript를 canon-model `Session`, `Run`,
`Event` record로 정규화합니다:

- **Session/Run**은 실행별 token 및 cost telemetry(모델, token 수, provider,
  원본 CLI가 보고하는 경우의 cost)를 담습니다.
- **Event**는 `token_usage` row와 **`user_directive`** event를 시간순으로
  섞어서 기록합니다. `user_directive`는 사용자 turn 메시지 하나당 하나씩
  생성되며, **내가 입력한 프롬프트의 전체 원문(verbatim text)**(`detail.text`)을
  담습니다 — 잘리거나 요약되지 않습니다. 이는 Hermes를 제외한 모든 adapter에
  적용됩니다. Hermes는 디스크 포맷에 turn 단위 메시지 테이블이 없어 directive
  텍스트를 읽어올 데이터가 없습니다 (Hermes는 항상 directive를 0건 생성합니다
  — 이는 문서화된 포맷상의 한계이지, 프라이버시 필터가 아닙니다).

별도의 "메타데이터만" 모드는 없습니다 — directive capture를 지원하는
adapter에서 세션을 ingest하면 프롬프트 텍스트도 함께 저장됩니다.

## 어디에 저장하는지

`canon init`의 기본 `canon.yaml`에서는 `session`, `run`, `event` record가
**`hot`** tier로 라우팅되며, 이는 내 레포의 `canon/hot.db`에 있는 **SQLite**로
백업됩니다. `canon init`은 `canon/hot.db*`(db 파일과 WAL/SHM 사이드카)를
`.gitignore`에 추가합니다 — 이 데이터는 Git에 커밋되지 **않으며**, 레포 히스토리에
전혀 들어가지 않습니다. `--watch`가 변경되지 않은 transcript를 다시 파싱하지
않도록 사용하는 파일 단위 ingest watermark는 같은 위치인 `canon/ingest/cursors/`
아래에 있으며, 이 역시 gitignore 대상입니다.

canon이 관리하는 다른 모든 record 종류(spec, evidence record, review,
divergence, strategy)는 **`local`** tier로 라우팅됩니다 — Git 기반이며
`canon/ledger` 아래에 커밋되어 일반적인 PR diff로 리뷰할 수 있습니다.
PostgreSQL(`hot`의 동급 대체제)과 S3/R2(`cold`)는 operator가 `canon.yaml`에
자격 증명을 명시적으로 추가할 때만 활성화되는 선택적 스케일업 backend입니다.

## 컴퓨터 밖으로 절대 나가지 않는 것

기본 `canon init` 설정에서는 **네트워크 egress가 전혀 없습니다**. 위에서 말한
`user_directive` 원문을 포함한 모든 record는 로컬 Git(`canon/ledger`, 커밋됨)과
로컬 SQLite(`canon/hot.db`, gitignore 대상, 디스크 전용)에만 머뭅니다. 어디로도
전송되지 않습니다.

데이터가 컴퓨터 밖으로 나가는 경우는 operator가 `canon.yaml`에 PostgreSQL DSN을
명시적으로 설정해 `hot`을 공유 Postgres 인스턴스로 교체하거나, S3/R2 버킷(`cold`)을
설정하는 경우 **뿐**입니다. 스캐폴딩된 기본 설정에는 둘 다 존재하지 않습니다.

## Redaction과 삭제

<Aside type="caution" title="아직 자동 redaction은 없습니다">
  Canon은 현재 ingest된 transcript에서 secret을 redact하거나 마스킹하거나
  스캔해 제거하지 **않습니다**. 입력한 프롬프트에 토큰, 비밀번호, 기타 민감한
  문자열이 포함되어 있었다면, 그 문자열은 `user_directive` event 텍스트에
  원문 그대로 저장됩니다. 이는 숨겨진 한계가 아니라 알려진 한계입니다 — 로컬
  transcript ingestion을 내 에이전트 세션의 다른 로컬 로그와 동일하게
  취급하시기 바랍니다.
</Aside>

현재 삭제는 수동이며 파일 단위로 이루어집니다:

- `canon/hot.db*`를 삭제하면 ingest된 모든 `session`/`run`/`event`
  record(모든 `user_directive` 텍스트 포함)가 사라지고 ingest cursor 상태도
  초기화됩니다.
- 없애고 싶은 `local` tier record가 있다면 `canon/ledger` 아래의 해당 record
  파일을 제거한 뒤, 다른 레포 변경 사항과 마찬가지로 커밋하면 됩니다.

자동 retention 정책이나 만료 기능은 없습니다 — 직접 제거하기 전까지 record는
계속 남아 있습니다.

## Project-scoped vs all-workspaces

기본적으로 `canon ingest sessions`는 **현재 project만** 스캔합니다: 레포의
메인 `git worktree` root와 연결된 모든 worktree(`git worktree list
--porcelain`으로 resolve)입니다. omp/pi와 Claude Code는 forward-encode된 cwd로
세션 저장소를 디스크에 분할해 두기 때문에, 범위 밖 project 디렉터리는 파일을
열기도 전에 pruning됩니다. Codex와 Hermes(디스크에서 분할되지 않음)는 대신
파싱 이후 row 단위로 기록된 workspace 기준으로 필터링됩니다.

`--all-workspaces`는 등록된 모든 transcript 홈 디렉터리를 대상으로 하는
machine-wide 스캔을 선택하는 옵션이며, project-scoping 이전의 동작으로
되돌립니다. **기본값(project-scoped)을 권장합니다** — 데이터 범위가 더 작을
뿐 아니라 더 유용합니다. `canon query --kind session`은 "이 컴퓨터에서 무슨
일이 있었는가"가 아니라 "이 project에서 무슨 일이 있었는가"에 답하도록
설계되었기 때문입니다.
