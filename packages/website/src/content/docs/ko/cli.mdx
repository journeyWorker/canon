title: CLI 레퍼런스
description: canon --help 출력을 기준으로 목적별로 묶은 전체 서브커맨드.
---

`canon`은 기능마다 하나의 서브커맨드를 갖는 단일 바이너리입니다. 언제든
`canon <command> --help` (또는 `canon <command> <subcommand> --help`)를
실행하면 정확한 플래그 목록을 확인할 수 있습니다 — 이 페이지는 그 출력을
그대로 반영합니다.

대부분의 서브커맨드는 `--repo <dir>` (기본값 `.`)를 받습니다: 리터럴
`.` 기본값은 `cwd`의 조상 디렉터리를 거슬러 올라가며 가장 가까운
`canon.yaml`을 찾고, 다른 명시적 경로는 그대로 사용되며 탐색을 하지
않습니다. 일부는 기본 사람이 읽기 좋은 표/요약 대신 기계가 읽을 수 있는
출력을 위한 `--json`도 받습니다.

## Try it

사전 설정 없이 바로 실행하는 데모 — gate가 막히고, evidence를 추가하면
통과하는 과정을 봅니다.

| Command | Description |
|---|---|
| `canon demo init [--repo <dir>]` | 일회용 evidence loop를 스캐폴드합니다: 실제 `canon init` 설정, `reviewer`를 요구하는 `policy.yaml`, 그리고 seed된 `dev` evidence 레코드 하나 — 그 결과 `canon gate check`는 `uncovered-cell auth.login.01`로 red 상태가 됩니다. 기존 `canon.yaml`은 덮어쓰지 않고 거부합니다. |
| `canon demo attest [--repo <dir>]` | 데모 scenario에 빠진 독립 `reviewer` evidence를 기록하여, 같은 `canon gate check`를 green으로 바꿉니다. |

## Setup

레포 설정을 스캐폴딩하고, 해석 결과를 확인하고, 동반 에이전트 스킬을
구체화합니다.

| Command | Description |
|---|---|
| `canon init [--repo <dir>]` | `<repo>/canon.yaml`에 동작하는 새 `canon.yaml` (`tiers:`/`routing:`/`specs:`/`plans:`)을 생성합니다. 기존 파일이 있으면 덮어쓰지 않습니다. |
| `canon init --check-config [--repo <dir>]` | 기존 `canon.yaml`을 읽기 전용으로 검증합니다. `inventory sync`/`ingest plans`/`tier age`가 쓰는 것과 동일한 strict loader를 그대로 체이닝합니다. |
| `canon context [--repo <dir>] [--json]` | 해석된 저작 표면을 출력합니다: 레코드 종류 + envelope 필드, enum 도메인, 조인 키 문법, 파티션 레이아웃, 정책 기반 요구사항, capability 버전. capability *쿼리*입니다 — `fmt --check`/`gate`가 같은 레포에서 진단을 보고하더라도 항상 0으로 종료합니다. |
| `canon skills install [--source <dir>] [--target <dir>]` | consumer 레포의 `.claude/skills/`와 `.codex/skills/`에 `canon/skills/<name>/SKILL.md`를 구체화하고, content-hash + 버전 락을 갱신합니다. 기본값: `--source canon/skills`, `--target .`. |

## Subjects

canon의 13번째 레코드 종류인 durable product unit(서브젝트)을 저작하고
라이프사이클을 통해 구동하며, import된 plan change를 그 아래로 adopt합니다.
subject-write-first 루프는 [캐논 개념](/ko/concepts/canon/#서브젝트)을
참고하세요. "feature docs"는 항상 Gherkin `.feature` 파일을, "subject"는
관리 레코드를 뜻합니다.

| Command | Description |
|---|---|
| `canon subject new <id> --domain <d> --title <t> [--summary <s>] [--owner-role <r>] [--repo <dir>] [--json]` | `Subject`을 저작합니다(kebab-slug `<id>`가 durable `subject_id` 조인 키). `--domain`은 vocabulary로 정의된 kebab slug입니다 — 기본 집합 `planning`/`design`/`dev`/`data`/`test`, `.canon/vocab`로 레포별 확장. 상태 `proposed`로 시작하며, `--owner-role`은 생략 시 `implementer`가 기본값입니다. 쓰기 시 envelope + 정책을 검증합니다. |
| `canon subject adopt <change_id> --subject <id> [--repo <dir>] [--json]` | import된 plan `Change`(`canon ingest plans` 결과)를 `Subject`에 연결합니다: `Change.subject_id`를 찍고 subject의 `change_ids`에 추가합니다. |
| `canon subject status <id> <state> [--repo <dir>] [--json]` | `proposed → specced → building → verifying → shipped`(및 any → `retired`)를 따르는 정책 게이팅 전이입니다. `verifying → shipped`는 연결된 모든 scenario가 최신 non-Divergent 판정을 가질 것을 추가로 요구합니다. 위반 시 failure class별로 출력하고 `1`로 종료하며 레코드는 변경되지 않습니다(fail closed). |

## Corpus

Gherkin `.feature` 스펙 코퍼스를 검증하고 확장합니다.

| Command | Description |
|---|---|
| `canon fmt --check [--repo <dir>] <ROOT>` | 코퍼스 루트(예: `spec/`)를 아티팩트 패밀리 스키마 + 레이아웃 디스크립터에 대해 검증합니다. `--check`가 fmt의 유일한 지원 모드입니다. |
| `canon inventory sync [--repo <dir>] [--spec-root <dir>]` | 설정된 `specs.roots[]` 항목마다 `canon-fmt`로 검증한 뒤, `(project_id, scenario_id)`당 하나의 `Scenario` ledger-index 레코드를 생성합니다. `--spec-root`는 하나의 ad hoc 루트로 `canon.yaml`을 완전히 오버라이드합니다. |
| `canon scenario new <TAG> --title <label> [--feature <path>] [--repo <dir>]` | `<area>.<surface>.<nn>` 태그에 대한 S11 준수 `.feature` 스텁을 추가합니다(없으면 생성). ledger 레코드는 쓰지 않습니다. |
| `canon feature new <SURFACE> --title <label> [--repo <dir>]` | 아직 시작되지 않은 `<area>.<surface>`를 위한 시나리오 없는 새 `.feature` 파일을 스캐폴딩합니다. 기존 파일이 있으면 덮어쓰지 않고 실패합니다. |

## Ingest

에이전트 CLI 세션, 증거 아티팩트, 외부 plan dialect를 티어드 스토리지로
적재합니다.

| Command | Description |
|---|---|
| `canon ingest sessions [--watch] [--interval-secs <n>] [--home <dir>] [--canon-yaml <path>] [--full] [--all-workspaces]` | 등록된 모든 세션 스토어 어댑터(`omp`/`pi`, Claude Code, Codex, Hermes)를 스캔하고, `Session`/`Run`/`Event` 레코드로 정규화한 뒤(사용자 턴에서 캡처한 `user_directive` 이벤트 포함) canon-store로 저장합니다. 기본값은 현재 프로젝트의 세션만 대상으로 합니다(메인 워크트리 + 연결된 모든 `git worktree` 루트); `--all-workspaces`는 머신 전체 스캔으로 되돌립니다. `--watch`는 파일 단위 워터마크와 함께 폴링합니다(기본 30초) — 변경된 트랜스크립트만 다시 파싱합니다. `--full`은 워터마크를 무시합니다. |
| `canon ingest artifacts [--watch] [--interval-secs <n>] [--repo <dir>] [--json]` | 조인 스파인의 아티팩트/판정 절반을 실행합니다: ledger/divergence/openspec-task 경로 어댑터와 handoff 레코드 소스 어댑터로 판정을 도출하고, regime 키가 매겨진 trajectory로 접어 canon-learn을 통해 저장합니다. `sessions`와 달리 반복된 `--watch` 패스는 아직 write-idempotent하지 않습니다. |
| `canon ingest plans [--dialect <name> --source <dir>] [--repo <dir>] [--json]` | 하나의 plan dialect — `openspec` change 디렉터리, 또는 `superpowers`의 writing-plans 형식 문서(s30: `# <title>`/`**Goal:**`/`### Task N:` 섹션, 섹션 안의 체크박스가 모두 체크되어야 해당 task가 done) — 를 canon-model의 Change/Task 레코드로 가져옵니다. 둘 다 같은 조인을 거쳐 `task_id = <change_id>#<n>`로 키가 매겨집니다. `--watch`는 없습니다 — plan은 operator가 직접 pull합니다. `--dialect`/`--source`는 `canon.yaml`의 `plans:` 섹션을 우회하는 일회성 import이며 둘 다 함께 필요합니다. |

## Trust

트러스트 스파인 gate와, 그것이 읽는 네이티브 `Review`/`Divergence` 판정
프로듀서입니다.

| Command | Description |
|---|---|
| `canon gate check [--repo <dir>] [--release]` | coverage/ledger/staleness/trust-ladder 체크 셋을 실행하고 위반 사항을 failure class별로 출력합니다. 종료 코드: `0` 클린, `1` gate-red, `2` 사용/로드 실패. `--release`는 release-scoped 트러스트 체크를 추가로 적용합니다. |
| `canon gate task <TASK_ID> [--repo <dir>]` | plan `task_id` (`<change_id>#<n>`) 하나에 대한 증거 게이팅된 체크박스 플립입니다. 증거가 없거나 형식이 잘못되었거나 조작된 경우 fail closed — 행은 플립되지 않은 채로 남습니다. |
| `canon gate promote [--repo <dir>] [--dry-run]` | `_staging/` 증거 레코드를 커밋된 ledger로 승급시키고, (role, surface)별로 단조 증가하는 `run_seq`를 부여합니다. `--dry-run`은 쓰지 않고 계획만 출력합니다. |
| `canon gate install-hooks [--repo <dir>] [--event <name>] [--matcher <m>] [--command <cmd>] [--timeout <secs>]` | `.claude/settings.json`/`.codex/hooks.json`에 멱등적인 diff-only hook-seam을 설치하고, 범용 pre-commit 스크립트도 함께 설치합니다. 기본값: `--event PreToolUse`, `--command "canon gate task"`, `--timeout 30`. |
| `canon gate selftest` | 번들된 fixture 코퍼스를 실행합니다: 발동이 증명된 모든 `FAILURE_CLASSES` 문자열을 `expected_failures.txt`와 정확히 일치시킵니다. `--repo`를 받지 않습니다. |
| `canon review add --project-id <id> --scenario-id <id> --reviewer <name> --pin <pin> --role <role> --upstream-ref <ref>` (또는 `--original-spec-ref <ref>`) `[--actor-id <id>] [--repo <dir>]` | 네이티브하고 귀속된 `Review` 레코드 하나를 씁니다. `--upstream-ref`/`--original-spec-ref` 중 정확히 하나가 필요합니다 — 둘 다 또는 둘 다 아니면 쓰기를 거부합니다. |
| `canon divergence stage --project-id <id> --scenario-id <id> --sha <sha> --reviewer <name> --role <role> [--status <status>] [--reason <r>] [--expiry <e>] [--round <n>] [--detail <d>] [--repo <dir>]` | `run_seq`가 없는 순서 없는 staging candidate 하나를 씁니다. `--status`는 `open`, `still-divergent`, `resolved`, `deferred` 중 하나를 받으며(`deferred`는 `--reason`/`--expiry`를 추가로 요구합니다), `run_seq`는 이후 `canon divergence promote`가 부여합니다. |
| `canon divergence promote [--repo <dir>] [--dry-run]` | 현재 staged된 모든 `Divergence` candidate를 일괄 승급시키고, `(project_id, role, surface)` 파티션 내에서 각각 단조 증가하는 `run_seq`를 부여합니다. |
| `canon divergence resolve --project-id <id> --scenario-id <id> --sha <sha> --reviewer <name> --role <role> [--round <n>] [--detail <d>] [--repo <dir>]` | `Resolved` candidate를 직접 커밋합니다 — staging 디렉터리를 건드리지 않습니다. |
| `canon divergence defer --project-id <id> --scenario-id <id> --sha <sha> --reviewer <name> --reason <r> --expiry <e> --role <role> [--round <n>] [--repo <dir>]` | `Deferred { reason, expiry }` candidate를 직접 커밋합니다 — staging 디렉터리를 건드리지 않습니다. |
| `canon divergence status [--repo <dir>] [--as-of <date>]` | divergence 번다운의 현재 상태 뷰입니다. `--as-of`는 `Deferred` 만료 평가 기준 시점을 지정하며, 생략하면 지금입니다. |

## Memory

증류된 전략, 역할+regime 범위의 retrieval, 그리고 dispatch 시점의 쓰기
경로입니다.

| Command | Description |
|---|---|
| `canon learn promote <STRATEGY_ID> [--repo <dir>] [--dry-run]` | 증류된 `StrategyItem` 하나(ULID 기준)를 `.canon/strategies/<role>/<id>.md`의 git 티어로 승급시키고, advisory promote lint(내용 길이, 리터럴 절대 경로)를 non-blocking 경고로 실행합니다. |
| `canon retrieve --role <role> (--regime <regime_key> \| --domain <d> [--subject <id>]) [--k <n>] [--repo <dir>] [--json]` | 역할+regime 범위의 전략 retrieval입니다. `--regime`을 직접 주거나 `--domain`/`--subject`로부터 도출합니다 — retrieval은 area 후보를 `<domain>-<subject_id>` 다음 `<domain>` 순서(폴백)로 시도합니다. `--role`은 `--regime`의 선행 세그먼트와 같아야 합니다(불일치 시 종료 코드 `2`). fail-soft: 그 전제조건이 성립하면 항상 `0`으로 종료하며, 스토어 장애는 빈 목록으로 저하됩니다. |
| `canon dispatch begin --role <role> --regime <regime_key> [--agent-id <id>] [--repo <dir>] [--json]` | `Run`(status `Running`)을 발급하고, `--role`/`--regime`에 대한 guidance를 retrieve하여 run의 `injected_guidance`에 기록한 뒤, 매니페스트를 `<repo>/.canon/dispatch/<run_id>.json`에 씁니다. |
| `canon regime-key --role <r> --repo <r> --area <a> --hash <h>` | 원본 세그먼트로부터 하나의 정규 `regime_key` (`<role>/<repo>/<area>/<hash>`)를 직렬화 + 검증합니다 — 모든 write/read 경로가 공유하는 Rust 직렬화기의 셸 대응물입니다. 키를 출력하고 `0`으로 종료하며, 형식이 잘못되면 `2`로 종료합니다. |

## Surface

현재 상태를 읽습니다: 리포트, 대시보드, ad hoc 쿼리, 티어 유지보수,
플러그인 오버레이, 셀프 체크입니다.

| Command | Description |
|---|---|
| `canon report [--repo <dir>] [--check] [--snapshot <dir>]` | markdown 상태 리포트를 생성하거나(기본값), 커밋된 사본과 byte-diff하거나(`--check`, drift 시 종료 코드 `1`), panel mart를 Parquet + `manifest.json`으로 export합니다(`--snapshot`, 둘 다 주어지면 우선). |
| `canon dashboard [--repo <dir>] [--snapshot <dir>] [--port <n>]` | 로컬에서 빌드된 대시보드 앱을 스냅샷 기준으로 서빙합니다. 기본 포트는 `4173`이며, `--snapshot`을 생략하면 매 실행마다 `.canon/dashboard-snapshot`에 새로 생성됩니다. |
| `canon query --kind <kind> [--since <ts>] [--repo <dir>] [--canon-yaml <path>] [--json] [--plugin <id>] [--change-id <id>] [--domain <d>] [--status <s>]` | 레코드 종류의 읽기를 그것이 있을 수 있는 모든 티어에 걸쳐 팬아웃하고 `at` 기준으로 병합합니다. `--plugin`은 해석된 플러그인의 오버레이 필드를 각 레코드에 투영하며, 미해석 플러그인이나 종류 불일치 시 fail-soft합니다. `--change-id`는 `--kind change`/`--kind task`에, `--domain`/`--status`는 `--kind subject`(도메인별 관리 뷰)에 범위를 제한하며, `--status`는 `--kind change`/`--kind task`에도 적용됩니다. |
| `canon tier age [--dry-run] [--repo <dir>] [--canon-yaml <path>]` | 모든 `canon.yaml` `aging:` 규칙을 한 번 적용해, 임계값을 넘은 레코드를 설정된 대상 티어로 이동시킵니다. `--dry-run`은 쓰지 않고 미리보기만 합니다. |
| `canon plugin sync <PLUGIN_ID> [--repo <dir>] [--spec-root <dir>]` | `<plugin-id>`의 매니페스트에 선언된 오버레이를 해석하고, 각각을 등록된 소스로 넘긴 뒤, 모든 candidate를 validate-then-write 파이프라인으로 씁니다. |
| `canon selftest [--json]` | 등록된 모든 크레이트의 fixture 코퍼스 셀프 테스트를 실행하고 EXPECTED oracle과 비교합니다. `canon gate selftest`와 별도로 추가됩니다. |
