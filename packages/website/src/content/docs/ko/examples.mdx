실제로 실행 가능한 네 가지 명령 시퀀스입니다. 한 줄짜리 요약은
[시작하기](/ko/getting-started/)를, 전체 플래그는
[CLI 레퍼런스](/ko/cli/)를 참고하세요.

## 1. 컨슈머 레포 연결하기

`canon.yaml`을 스캐폴드한 다음 스펙 코퍼스를 검증하고 인덱싱합니다.

<Steps>

1. **설정 파일 스캐폴드.**

   ```bash
   bunx @journeykit/canon init
   ```

   바로 동작하는 `canon.yaml`을 씁니다. `canon init`은 내구성 있고
   리뷰 가능한 레코드에는 Git을, 대량으로 발생하는 운영 레코드에는
   로컬 SQLite를 사용합니다. PostgreSQL과 S3 호환 스토리지는 선택적
   스케일업 백엔드입니다. canon-model의 12개 레코드 종류 중 7개는
   환경 변수가 필요 없는 `local`(git 백엔드) rung으로 라우팅되고,
   `task`, `handoff`, `session`, `run`, `event`는 `hot`(기본값
   SQLite, `canon/hot.db`, gitignore 처리됨)으로 라우팅됩니다.
   주석 처리된 `postgres`/`s3` 스탠자는 스케일업 전환 방법을
   보여주며, 실제로 동작하는 `specs.roots[]` 항목 하나와 비어 있는
   `plans: { sources: [] }`도 포함됩니다:

   ```yaml
   tiers:
     local:
       backend: git
       root: canon/ledger
     hot:
       backend: sqlite
       path: canon/hot.db
     # swap hot to postgres once you need a real server:
     #   backend: postgres
     #   dsn_env: CANON_PG_DSN
     #   schema: canon_v1
     # cold:
     #   backend: s3
     #   bucket_env: CANON_R2_BUCKET
     #   prefix: "canon/"
   routing:
     change: local
     task: hot
     scenario: local
     session: hot
     run: hot
     event: hot
     handoff: hot
     review: local
     divergence: local
     trajectory: local
     strategy_item: local
     evidence_record: local
   specs:
     roots:
       - id: root
         root: specs
   plans:
     sources: []
   ```

   `canon init`은 기존 `canon.yaml`을 덮어쓰지 않고 거부합니다.
   기존 설정은 언제든 다시 검증할 수 있습니다 (`inventory sync` /
   `ingest plans` / `tier age`가 쓰는 것과 동일한 세 개의 엄격한
   로더를 연쇄 실행):

   ```bash
   bunx @journeykit/canon init --check-config
   ```

2. **스펙 코퍼스 검증.** `.feature` 파일은 `specs.roots[]`가
   가리키는 경로(기본값 `specs/`) 아래에 있습니다:

   ```bash
   bunx @journeykit/canon fmt --check specs
   ```

   코퍼스를 canon의 아티팩트-패밀리 스키마와 레이아웃 디스크립터에
   대해 검사합니다. provenance 누락, 레이아웃 문법 위반 등 어떤
   위반이든 보고되며 — 여기가 깨끗해지기 전에는 다음 단계가
   실행되지 않습니다.

3. **레저 인덱스 구체화.**

   ```bash
   bunx @journeykit/canon inventory sync
   ```

   설정된 각 root에 대해 동일한 `fmt --check` 검사를 다시 수행한
   뒤, `<root>/features/**/*.feature`를 순회하며 `(project_id,
   scenario_id)`쌍마다 하나의 `Scenario` 레저 레코드를 upsert
   합니다. 위반이 발생하면 해당 root 전체가 중단됩니다 — 부분
   sync는 절대 없으며, 그 root에 대해 레코드가 하나도 쓰이지
   않습니다. 변경되지 않은 `.feature` 파일(동일한 `source_digest`)은
   no-op이고, 변경된 파일은 새 레코드를 추가합니다(append-only,
   기존 레코드는 절대 덮어쓰지 않음).

</Steps>

## 2. 증거 루프

시나리오를 작성하고, 리뷰어의 판정을 붙이고, 게이트를 통과시킨 뒤
상태 리포트를 렌더링합니다.

<Steps>

1. **시나리오 스텁 스캐폴드.**

   ```bash
   bunx @journeykit/canon scenario new auth.login.01 --title "Login rejects bad password"
   ```

   S11 규격을 만족하는 `.feature` 스텁을 추가합니다 — 파일이 없으면
   새로 만듭니다 — 대상 경로는 태그 자체의 `<area>.<surface>`에서
   유도되거나(또는 설정된 `specs.roots[]` 항목 아래의 명시적
   `--feature <path>`), 레저 레코드는 쓰지 않습니다. Gherkin
   본문은 직접 채워 넣은 다음, 인덱싱을 위해 1번 워크스루의
   `fmt --check`와 `inventory sync`를 실행하세요.

2. **리뷰 부착.**

   ```bash
   bunx @journeykit/canon review add \
     --project-id root \
     --scenario-id auth.login.01 \
     --reviewer alice \
     --pin "$(git rev-parse HEAD)" \
     --original-spec-ref specs/auth.feature \
     --role dev
   ```

   귀속(attributed)된 native `Review` 레코드 하나를 씁니다.
   `--upstream-ref` / `--original-spec-ref` 중 정확히 하나가 필요합니다
   — provenance는 절대 합성되지 않습니다.

3. **작업 게이트 통과.**

   ```bash
   bunx @journeykit/canon gate check
   ```

   증거 코퍼스에 대해 coverage / verdict-ledger / staleness /
   trust-ladder 체크 세트를 실행하고, 모든 위반을 실패 클래스별로
   그룹화해 출력합니다(`uncovered-cell`, `stale-evidence`,
   `flagged` 등). 클린이면 exit `0`, gate-red면 `1`, 사용법/로드
   실패면 `2`입니다. `--release`를 추가하면 release-scoped
   `trust-below-required` 체크도 함께 활성화됩니다 — 전체 실패
   클래스 표는 [트러스트 스파인](/ko/concepts/trust-spine/)을
   참고하세요.

4. **상태 리포트 렌더링.**

   ```bash
   bunx @journeykit/canon report
   ```

   다섯 개 상태 패널(트러스트 매트릭스, 세션 비용, 역할별 메모리,
   플라이휠 퍼널, 리뷰 번다운)을 `canon/REPORT.md`에 렌더링합니다
   — 생성된 파일이며 절대 손으로 편집하지 않습니다. `canon gate
   check`를 CI에 연결하는 것과 같은 방식으로 `canon report --check`를
   연결해 커밋된 파일을 현재 입력과 byte-diff하세요.

</Steps>

## 3. 세션 텔레메트리

에이전트 CLI 트랜스크립트를 티어드 스토리지로 적재한 뒤 다시
읽어봅니다.

<Steps>

1. **한 번 적재하고, 이후 watch.**

   ```bash
   bunx @journeykit/canon ingest sessions
   bunx @journeykit/canon ingest sessions --watch   # 30초마다 폴링 (--interval-secs)
   ```

   등록된 모든 어댑터(`omp`, `hermes`, `claude`, `codex`)를
   스캔하여 `Session` / `Run` / `Event` 레코드로 정규화하고,
   `canon query`가 쓰는 것과 동일한 티어 해석을 통해 적재합니다.
   소스별 워터마크 커서가 지난 실행과 바이트 단위로 동일한
   소스는 건너뛰므로 — 정상 상태의 `--watch` 루프는 실제로
   변경된 것만 다시 파싱합니다. `--full`로 전체 재스캔을 강제할
   수 있습니다.

2. **레코드 다시 조회.**

   ```bash
   bunx @journeykit/canon query --kind session --json
   bunx @journeykit/canon query --kind run --since 2026-07-01
   ```

   해당 종류가 존재할 수 있는 모든 티어에 걸쳐 읽기를 팬아웃하고
   `at` 기준으로 병합합니다. `--json`은 병합된 레코드 본문을
   출력하고, 생략하면 사람이 읽기 좋은 테이블로 출력합니다.

3. **대시보드 열기.**

   ```bash
   bunx @journeykit/canon dashboard
   ```

   새 Parquet 스냅샷을 생성하고(내부적으로 `canon report
   --snapshot`을 `canon/dashboard-snapshot`에 실행) 빌드된
   `packages/dashboard` 앱을 로컬에서 그 스냅샷을 가리켜
   서빙합니다 — `canon report`가 렌더링하는 것과 동일한 다섯 개
   패널을, 브라우저에서 다시 계산하는 대신 스냅샷에서 읽습니다.
   다른 곳에서 생성한 스냅샷(예: 다운로드한 CI 아티팩트)을
   그대로 서빙하려면 `--snapshot <dir>`을 넘기세요.

</Steps>

## 4. superpowers 플랜 가져오기

superpowers `writing-plans` 플랜 문서를 `Change`/`Task` 코어
레코드로 미러링합니다 — 손으로 다시 작성할 필요가 없습니다.

<Steps>

1. **소스 설정.**

   ```yaml
   plans:
     sources:
       - dialect: superpowers
         root: docs/superpowers/plans
   ```

   또는 `canon.yaml`을 우회하는 일회성 가져오기:

   ```bash
   bunx @journeykit/canon ingest plans --dialect superpowers --source docs/superpowers/plans
   ```

2. **가져오기 실행.**

   ```bash
   bunx @journeykit/canon ingest plans
   ```

   root 아래의 각 `*.md` 플랜 문서가 하나의 `Change`가 됩니다:
   `change_id`는 슬러그화된 파일명 stem이며(날짜 접두어는 그대로
   유지), `summary`는 `**Goal:**` 줄의 텍스트입니다. 각
   `### Task N: <name>` 섹션은 하나의 `Task`가 됩니다
   (`task_id = <change_id>#<N>`), 섹션에 체크박스가 하나 이상
   있고 전부 체크되어 있으면 `done`, 그렇지 않으면 `open`입니다 —
   체크박스가 없는 섹션은 항상 `open`이며 절대 `done`이 되지
   않습니다. Step과 산문 헤딩은 절대 읽히지 않습니다.

3. **미러 다시 읽기.**

   ```bash
   bunx @journeykit/canon query --kind change --json
   bunx @journeykit/canon query --kind task --json
   ```

   가져온 레코드는 평범한 코어 레코드입니다 — `canon query`는
   손으로 작성한 `canon gate task`와 구별할 수 없으며(구별할 수
   없어야 합니다). 이 미러는 조회 가능할 뿐 절대 권위 있는
   소스가 아닙니다: `canon gate check`는 증거(`Review`/
   `EvidenceRecord`)만 읽을 뿐, 플랜에서 가져온 `Change`/`Task`
   레코드는 절대 읽지 않습니다 — 플랜 체크박스를 뒤집는 것만으로는
   게이트를 red로도 green으로도 바꿀 수 없습니다.

</Steps>

## 5. 서브젝트-도메인 루프

서브젝트를 저작하고, import된 change를 그 아래로 adopt하고,
서브젝트 스코프 가이던스를 꺼낸 뒤, 증거 게이팅된 라이프사이클을
통해 구동합니다.

<Steps>

1. **서브젝트를 먼저 저작.**

   ```bash
   bunx @journeykit/canon subject new checkout-revamp \
     --domain dev --title "Checkout revamp" --owner-role implementer
   ```

   kebab-slug `<id>`가 durable `subject_id` 조인 키이며, 서브젝트는
   상태 `proposed`로 시작합니다. `--domain`은 vocabulary slug입니다
   (기본 `planning`/`design`/`dev`/`data`/`test`).

2. **import된 plan change를 그 아래로 adopt.**

   ```bash
   bunx @journeykit/canon subject adopt checkout-revamp-plan --subject checkout-revamp
   ```

   `Change.subject_id`를 찍고 change를 서브젝트의 `change_ids`에
   추가합니다 — plan 단위가 이제 관리되는 product unit에 속합니다.

3. **작업 전 서브젝트 스코프 가이던스 retrieve.**

   ```bash
   bunx @journeykit/canon retrieve --role dev --domain dev --subject checkout-revamp
   ```

   retrieval은 area 후보를 `dev-checkout-revamp` 다음 `dev` 순서(폴백)로
   시도하므로, 서브젝트가 새것일 때는 도메인의 공유 메모리로 폴백하고,
   그렇지 않으면 서브젝트 자신이 축적한 교훈이 우선합니다.

4. **라이프사이클 구동.**

   ```bash
   bunx @journeykit/canon subject status checkout-revamp specced
   bunx @journeykit/canon subject status checkout-revamp building
   bunx @journeykit/canon subject status checkout-revamp verifying
   bunx @journeykit/canon subject status checkout-revamp shipped
   ```

   `verifying → shipped` 단계는 증거 게이팅됩니다: 서브젝트가 연결한
   모든 scenario가 최신 non-Divergent 판정을 가질 때에만 성공합니다.
   그렇지 않으면 문제되는 scenario를 출력하고 `1`로 종료하며 상태는
   `verifying`으로 유지됩니다.

5. **도메인별 관리 뷰 읽기.**

   ```bash
   bunx @journeykit/canon query --kind subject --domain dev
   bunx @journeykit/canon report
   ```

   `canon report`의 Subjects 패널이 도메인별로 각 서브젝트의 상태 ×
   scenario 커버리지를 롤업합니다.

</Steps>

## 사례 연구: Red Panda Ridge

[`examples/platformer`](https://github.com/journeyWorker/canon/tree/main/examples/platformer)
는 canon을 도그푸딩하는 다섯 개의 페르소나 에이전트가 — canon의
각 역할(voice)마다 한 에이전트씩 — 실제 CLI를 사용해 처음부터
끝까지 만들고 검증한 작은 Vite + React + PixiJS 플랫포머입니다:

| Voice | 한 일 | canon 표면 |
|---|---|---|
| planning | 구현 플랜 작성 (superpowers `writing-plans` 포맷) | 플랜 문서 → `canon ingest plans --dialect superpowers`로 미러링 |
| design | 새 behavior 시나리오 8개 작성 | `canon context` → `canon scenario new` × 8 → `canon fmt --check` (위반 0건) |
| dev | 16개 시나리오 레저를 대상으로 구현, 플랜 체크박스 반영 | `canon query --kind scenario`, `canon retrieve --role dev` |
| test | 16개 시나리오 전부 독립적으로 검증, 증명 가능한 결함 2건 수정 | `canon review add` × 16 (attested, 검증된 sha에 고정) |
| review | 확인된 ship-blocker 8건을 divergence로 기록 | `canon divergence stage`/`promote` × 8 → `resolve` × 6 |

플랜 문서는 로드맵이고, (`canon inventory sync`로 구체화된)
16개 시나리오 `.feature` 코퍼스가 dev와 test가 실제로 작업한
승인 권위(acceptance authority)입니다 — 이 둘은 절대 합쳐지지
않습니다:

```
dev claims done   → plan checkboxes → canon ingest plans   (mirror, never trusted)
test proves done  → canon review add per scenario           (evidence)
review disagrees  → canon divergence stage/promote          (tracked, run_seq-ordered)
fix lands         → re-verify → attest → divergence resolve (burndown to zero)
                  → canon gate check: clean (0 violations)
```

최종 상태: 16/16개 시나리오 attested, divergence burndown
`open=0 resolved=6`, `canon gate check` 클린.
