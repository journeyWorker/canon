# canon-report-dashboard

S9 (`s9-unified-surface`) is canon's single status surface: `canon report`
renders five panel marts (already computed by `canon-store`'s DuckDB view
layer — `crates/canon-store/sql/views.sql`) to markdown, and `canon
dashboard` serves a small zero-network web app that renders the SAME
numbers from a Parquet snapshot of those same views. Neither tool
computes anything itself — `crates/canon-report` is a thin
render/export layer over the views, and `packages/dashboard` is a thin
SELECT layer over the snapshot (design D1/D3: exactly one place
aggregate state is computed, never a second aggregation layer in Rust
or in the browser).

## Generated, never hand-edited (design D16)

`canon/REPORT.md` carries a leading `<!-- GENERATED by \`canon report\`
… DO NOT EDIT. -->` comment. If it looks stale or wrong, the fix is
**always** re-running `canon report` (or fixing the underlying
ledger/policy/mart it renders) — never hand-editing the file. The
report header embeds ONLY input-derived digests (corpus hash, policy
hash, ledger-head hash) and **no timestamp, no `source_git_sha`**
(decision 11): a committed report is reproducible from named inputs,
so `canon report --check` can byte-diff it deterministically. If you
see a `source_git_sha` field anywhere in `canon/REPORT.md`, that is a
bug — it belongs only in `--snapshot`'s `manifest.json` (below), which
is never drift-checked.

The dashboard is the same discipline one layer further: it has zero
mart-derivation code of its own (`packages/dashboard/src/panels/*.ts`
are each one bare `SELECT … FROM mart_x ORDER BY …`), so it can never
render numbers that contradict `canon report`'s.

## `canon report [--repo <dir>]`

Renders the current five-panel report to `<repo>/canon/REPORT.md`
(configurable — `crates/canon-cli/src/report.rs`'s `resolve_roots`
reads `canon.yaml`'s `tiers.git.root` for the git-tier root, falling
back to `canon/ledger`). `--repo` resolves through the same
nearest-`canon.yaml`-ancestor walk every other subcommand uses (design
D7) — omit it (or pass `.`) to run from any subdirectory of a canon
repo.

```bash
canon report                 # writes canon/REPORT.md
canon report --repo /path/to/repo
```

## `canon report --check` — the CI drift gate

Regenerates the report **in memory** and byte-compares it against the
committed `canon/REPORT.md` (design D2, `parity.py cmd_report` lifted
near-verbatim):

- `MISSING` (exit 1) — no report has ever been written; run `canon
  report` first.
- no drift (exit 0) — the committed file is byte-identical to what the
  current inputs would render right now.
- `DRIFT` (exit 1) — the committed file disagrees with current inputs
  (someone hand-edited it, or the underlying corpus/policy/ledger moved
  and nobody re-ran `canon report`).

```bash
canon report --check   # wire this into CI exactly like `canon fmt --check`/`canon gate check`
```

Wire it as a CI step (or a pre-push hook) the same way `canon gate
install-hooks` wires `canon gate check` — this skill adds no second
installer; run `canon report --check` directly wherever your CI already
runs `canon gate check`/`canon fmt --check`.

## `canon report --snapshot <dir>`

Exports the same five views to `<dir>/<table>.parquet` (one explicit
`COPY "<table>" TO '<table>.parquet' (FORMAT parquet)` per table, never
`EXPORT DATABASE` — design D3's own digit-escaping-bug fix) plus a
declared `<dir>/manifest.json`:

```json
{
  "generated_at": "...",
  "source_git_sha": "...",
  "source_digest": "...",
  "tables": [
    { "table": "mart_trust_matrix", "file": "mart_trust_matrix.parquet" },
    { "table": "mart_session_costs", "file": "mart_session_costs.parquet" },
    { "table": "mart_role_memory", "file": "mart_role_memory.parquet" },
    { "table": "mart_flywheel_funnel", "file": "mart_flywheel_funnel.parquet" },
    { "table": "mart_review_burndown", "file": "mart_review_burndown.parquet" }
  ]
}
```

This IS where `source_git_sha`/`generated_at` live (never in the
drift-checked markdown header, above) — a snapshot is inherently
output-inclusive provenance, unlike a committed generated file.

```bash
canon report --snapshot /tmp/canon-snap
```

## `canon dashboard [--repo <dir>] [--snapshot <dir>] [--port <n>]`

Serves the already-built `packages/dashboard` static app locally,
pointed at a real snapshot via the app's own `?snapshot=` override:

```bash
# build the app once (from the repo root, or inside packages/dashboard)
bun install && bun run --cwd packages/dashboard build

canon dashboard                       # regenerates a fresh snapshot every run, serves it
canon dashboard --snapshot /tmp/canon-snap --port 8080
```

- `--snapshot` omitted: `canon dashboard` calls `canon report
  --snapshot` itself, every run, into the conventional
  `canon/dashboard-snapshot` scratch directory (a sibling of
  `canon/REPORT.md`/`canon/ledger`/`canon/r2`/`canon/learn`) — always
  fresh, no separate "last snapshot" state to track.
- `--snapshot <dir>` given: served AS-IS when `<dir>` already has a
  `manifest.json` (e.g. a snapshot you generated separately, or a
  downloaded CI artifact) — never silently regenerated over it. If
  `<dir>` has no `manifest.json` yet, one is generated there once.
- Prints the URL to open — `http://127.0.0.1:<port>/?snapshot=/live-snapshot/`
  — then serves until you stop it (Ctrl-C). The `packages/dashboard`
  app itself is never rebuilt or modified by this command; if it is
  missing (no `packages/dashboard/dist/index.html`), `canon dashboard`
  fails with the exact build command to run.

The dashboard's freshness banner renders `manifest.json`'s
`generated_at`/`source_git_sha`/`source_digest` with an explicit
"scoped to this snapshot, not necessarily the live checkout" note
(design Risk section) — treat "matches `canon report`" as "matches the
inputs that produced THIS snapshot," not "matches HEAD right now."

## Reading the five panels

Every panel is a bare, already-fully-aggregated `SELECT` over one
`canon-store` mart (`crates/canon-store/sql/views.sql`) — no
client-side grouping/summing anywhere.

- **Trust matrix** (`mart_trust_matrix`) — change/task × `covered` ×
  `green` × `who`. `covered` = required evidence exists per policy;
  `green` = the latest verdict actually passed.
- **Session costs** (`mart_session_costs`) — token/cost by
  role/repo/session, `session_id`-keyed.
- **Role memory** (`mart_role_memory`) — strategies, hit rate
  (active/total), and an effect proxy (avg. source-trajectory count)
  per role namespace, over `canon-learn`'s `StrategyItem` store.
- **Flywheel funnel** (`mart_flywheel_funnel`) — verdicts → distilled →
  retrieved → applied counts. A large drop between any two stages is
  the actionable signal (e.g. verdicts landing but nothing ever gets
  distilled into a strategy).
- **Review burn-down** (`mart_review_burndown`) — per-day
  evidence/divergence counts plus a running "divergences still open"
  total. A running total that never trends down means feedback isn't
  being closed out.

## What this skill does NOT cover

- `canon-store`'s own `stg_/int_/mart_` view layer (S2) or how the
  five S9-owned marts are computed — see `crates/canon-store/sql/
  views.sql` and the `tiered-storage` skill.
- `canon gate`'s evidence-gated checkbox flip / trust ladder — see the
  `trust-spine-gate` skill. `canon report`'s trust-matrix panel READS
  the same evidence corpus `canon gate check` validates, but never
  gates or flips anything itself.
- `canon-learn`'s strategy distillation/promotion — see the
  `canon-reward` skill; the role-memory panel is a read-only view over
  that store.
- Building/deploying `packages/dashboard` for a hosted (non-local)
  audience — `canon dashboard` is a local dev-server convenience; a
  hosted deployment would serve the same static `dist/` output behind
  whatever web server/CDN the operator already runs, with a snapshot
  refreshed on a schedule (e.g. `canon report --snapshot` in CI,
  publishing the result) — not built by this skill or `canon-cli`
  itself.
